## 背景与运行环境

- 这是用 LLM 自动设计调度规则的**动态柔性装配流水车间**排产系统。四步主流程：
  **① 导入数据 → ② 仿真 → ③ 优化求解 → ④ 结果评审**。
- 真实数据规模（用户本次输入）：订单 516 / 任务令 4,363 / 工序 16,536 / 机器 1,149 /
  工装 0 / 人员 0。这是**超大实例**，很多"小数据正常、大数据崩"的 bug 会在此规模暴露。
- 输入文件：`instance_output.xlsx`（用户真实数据）。后端导入接口按 sheet 名解析，**文件名无关**，
  只要内容 sheet 正确即可。
- 运行环境：项目自带 `.venv`（含 ortools / numpy / openpyxl / fastapi）。无需 LLM API Key（无则模板回退）。

各步骤的**后端 API**（用 curl / python requests 做确定性端到端测试，比手点 UI 更可靠）：

| 步骤 | 方法 / 端点 | 关键入参 | 返回里要验证的字段 |
|---|---|---|---|
| ① 导入 | `POST /api/instance/import-excel`（multipart，`file` 字段） | `instance_output.xlsx` | `status=="ok"`、`summary`（orders/tasks/operations/machines 数量）、`validation.status`、`validation.error_count`（应为 0） |
| ② 仿真 | `POST /api/simulate`（`SimReq`：`rule` 如 `"ATC"`） | 规则名 | `gantt`（工序排程列表）、`feasible`、`kpi`/`metrics`、`diagnosis`（不可行时有原因） |
| ③ 优化 | `POST /api/optimize/hybrid`（`HybridConfig`）→ 拿 `task_id`；轮询 `GET /api/optimize/hybrid/status/{task_id}`；取 `GET /api/optimize/hybrid/result/{task_id}` | objectives / generations / time_limit_s | status 里的 `approximate_evaluations` / `exact_evaluations` / `coarse_pool_size` / `feasible_ratio` 四张卡片；result 里要有可行解 |
| ④ 评审 | 前端"方案评审"页（`solution-review`）：对比候选（`renderCandidateMetricMatrix`）、精确冠军参考（`/optimize/exact-reference`）、AI 评审建议；保存 `PUT /api/workflow/review` | 选中的 solution_id 列表 | 候选能列出、指标矩阵有值、可导出 Excel（`/optimize/hybrid/export-solution`） |

> 注意：③ 优化在你这个规模下**极慢**（已知单步离散事件仿真在 1.6 万工序/1149 机器下要分钟级，

> 能产出可行解；性能专项优化后再做"真实时长"复测。

## 任务二：已知问题的实现计划（用户明确点名，只出计划不改代码）

### A. 甘特图必须正确呈现"资源不可用"和"排产情况"，并清晰展示工序时间段

当前实现 `frontend/app_v2.js` 的 `renderTimeline()`（约 1498 行）存在以下问题，需**给出修复实现计划**：

- **资源不可用呈现不足**：图例虽写了"班次外/计划停机/非计划停机"，但机器行的"不可用时间段"
  （来自 `/downtime` 接口与 `machine.downtimes`，代码里 `getMachineDowntimes` 约 1006 行、
  机器时间线 overlay 约 1351–1367 行）在超大实例下**没有清晰、稳定地画在甘特上**。要求：
  - 每台机器行上，把停机（计划/非计划）与班次外时间画成**明确可辨的色带/斜纹带**，与"排产条"
    视觉区分；鼠标悬停显示起止与类型。
  - 停机数据要真正来自后端的 downtime 存储，不要只画图例不画带。
- **排产时间段不清晰**：每道工序在机器上的占用应是一条**带明确起止时刻的条块**，标注工序号/
  订单/任务；已完成/进行中/未来三态用颜色区分（已有 `status-completed/processing/future`，保留）。
  确认条块时间轴与 `gantt` 返回的 `start/end` 一致，缩放/平移后时刻仍准确。
- **超大实例被静默截断，看不到资源全貌**：`CONFIG.GANTT_MAX_GROUPS=40`
  （app_v2.js:11、截断逻辑 1440–1446、1490），你的实例有 **1149 台机器**，甘特只显示排程最多的
  前 40 台，**其余 96% 机器不可见**——这正是"不能很好呈现资源情况"的主因。要求：
  - 不要直接砍掉 96% 资源。改为：提供**资源维度筛选/分页/聚合**（如按机器类型、按是否含停机、
    按时间窗、按订单聚焦），让用户能按需看到关心的资源；或做虚拟滚动/分图。
  - 至少保证"含停机的机器"和"当前选中订单涉及的机器"优先展示，而不是只看排程最多的前 N 台。

### B. 订单下拉选择后必须可筛选

当前 `renderTimeline` 已有订单下拉 `data-gantt-order-select`（约 1511–1536 行），选中后用
`visibleEntries` 按 `order_id` 过滤（1515–1517）。但用户要求"下拉之后**要可以筛选**"，
说明当前筛选能力不够（可能：下拉只切订单、无法在订单内再筛；或大实例下被强制单订单、
过滤不生效/不重渲染）。要求：

- 选中某个订单后，提供**进一步的筛选控件**（例如：按工序状态、按机器/资源、按时间窗、按关键字
  搜索工序号），筛选实时作用于甘特视图。
- 确保订单下拉变更后**视图确实重新过滤并重渲染**（检查 `data-gantt-order-select` 的 change 事件
  与 `app.ganttOrderFilter[id]` 的联动）。
- 小实例的"全部订单"入口保留；大实例（> `GANTT_ALL_ORDERS_MAX_OPS=2000`，见 13/1510 行）仍允许
  先选订单再筛，不要因规模直接锁死为只读单订单。

### A/B 计划粒度要求

对每个修复点，实现计划必须包含：

1. **根因定位**：文件路径 + 具体函数/行号 + 一段**现有相关代码摘录**（证明你真的读过该处）。
2. **具体改动**：以**「新增 / 替换 / 删除」三种形式之一**给出**可直接粘贴的完整代码块**，精确到：
   - 落在哪个**文件**、哪个**函数/方法**、锚定到哪个**行号区间**；
   - 若"替换"，先给「替换前（Before）」再给「替换后（After）」完整代码，不要只给 diff 摘要；
   - 若"新增"，给出要新增的**函数/代码块全文**及其**插入位置**（在哪个函数之后/哪个标记之前）；

## 任务三：性能优化实现计划（规模 16k 工序时过慢，只出计划不改代码）

对你这个规模，仿真和优化都偏慢。先用**计时**量化每一步墙钟耗时，定位瓶颈，再给出**最小必要、
精确到代码**的优化实现计划，并用「优化前 vs 优化后」的计时预期说明收益。已知高概率瓶颈
（供参考，以实测为准）：

- **仿真派工全扫描**（`core/simulator.py` 的 dispatch 路径）：每次派工遍历某工艺类型**全部**就绪
  工序再按 `machine.id in eligible` 过滤，缺"机器→可派工序"反向索引；每完成一道工序还遍历全部
  机器筛 IDLE。规模下乘积极大。优化方向：加 `machine_ready_ops[machine_id]` 与 `machines_by_type`
  反向索引，把扫描降到实际候选。
- **每候选重建 features 字典**（`core/simulator.py` `_features()`）：缓存不随 `now` 变化的部分，
  减少每候选 dict 分配。
- **优化求解精确评估过慢**（`optimization/hybrid_nsga3_alns.py` 的 `run_exact_simulation` 调用）：
  单候选离散事件仿真分钟级，导致 `exact_evaluations` 长时间不增长。可结合上面仿真优化、
  以及 `optimization/parallel_eval.py` 的进程池并行度调优。

性能计划粒度要求：

1. **瓶颈定位**：每个瓶颈点给出 文件:函数:行号 + 现有代码摘录 + 计时数据（你实测的 before 耗时）。
2. **具体优化代码**：以上述「新增/替换/删除 + 直接可粘贴代码块 + 锚定行号」的形式给出改动。
3. **验收预期**：写明"实施后在同实例上的预期 before/after 对照"（如仿真从 X 分钟降到 Y 分钟），
   **不得为提速改变算法语义或牺牲可行性**。
