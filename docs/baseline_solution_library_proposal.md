# 基准方案库（Baseline Solution Library）设计提案

> 状态：决策已定，实施方案细化中（仍仅方案，未实现）
> 目标：把"夜间对实际车间数据做三维多目标大规模优化"产出的若干派工参数方案持久化为**基准方案库**；白天优化与演示默认从这些方案**热启动**，从而同时解决两件事——① 优化器起点不再是"仅可行"的 ATC，而是贴近实际数据的高质量先验；② 快速演示一打开就是好结果，而非最简规则的结果。

---

## 1. 动机与核心洞察

你的系统里，**一个被优化的解 = 一组派工特征权重向量（`feature_weights`，16 维）**。优化器搜出来的每个 Pareto 解，本身就是"一条派工规则"。

因此"整理出更好的基线规则并固化"不需要另起炉灶，本质就是：

> **用优化器当"规则设计器"，把它的若干均衡/极端解，蒸馏、验证、持久化成一个方案库；后续优化与演示直接以这些解热启动。**

这带来两个收益：
- **优化起点增强**：种群从"高质量先验"起步，收敛更快、前沿整体抬升。
- **演示体验增强**：仿真/评审界面默认规则不再是最简规则，而是贴近实际数据的强方案。

一条权重向量优化的是"派工策略"而非某张具体排程；只要白天实例与夜间快照结构相近（你确认过"最多订单只有一点点小调整"），夜间学到的策略在白天仍接近最优——所以把它当 **warm start prior** 是标准且稳健的做法。

## 2. 目标集定义（三维）

你指定的三个目标，对齐到系统真实存在的目标键：

| 你的表述 | 系统目标键 | 方向 | 是否在 `OBJECTIVE_SPECS` 中 |
|---|---|---|---|
| 关键设备活跃窗口利用率 | `critical_active_window_utilization` | max | ✅ 现成（objectives.py:91） |

| 总延误订单比例 | `main_order_tardy_ratio`（**已定：用现成键**） | min | ✅ 现成（objectives.py:67） |

**目标键已定（评审拍板）：使用现成的 `main_order_tardy_ratio`（主订单延误率，objectives.py:67，min）。**

说明：系统当前无"全部订单延误占比"键，相关现成键为 `main_order_tardy_ratio`（仅主订单）、`tardy_job_count`（任务数）、`total/avg/max_tardiness`（时长）。本次采用 `main_order_tardy_ratio`——**零代码改动**，其余设计不变。若未来需要"全部订单比例"，可再行新增 `tardy_order_ratio`（只加不减），不影响本方案。

## 3. 整体架构（数据流）

```
┌─────────────── 夜间（冷搜索，大规模、离线）───────────────┐
│  输入：当天实际车间快照（planning_context / inst_* 表）      │
│  跑 HybridNSGA3ALNSOptimizer（目标 = 上述三维，限时间/代数） │
│       → 产出 Pareto 近似前沿                                │
│  从前沿自适应抽取方案（极端 + 膝点 + 簇代表，去重）          │
│  质量门禁：每个方案须稳定优于 ATC，否则丢弃                 │
│  持久化 → baseline_solutions 表（status=active）            │
└──────────────────────────────────────────────────────────┘
        │  (库可空、可失效、可回退，见 §7)
        ▼
┌─────────────── 白天（热启动，小规模）────────────────────┐
│  优化器 _seed_population 在现有种子基础上追加库里所有        │
│  active 方案作为种子 → 很短代数即收敛                       │
│  可选：用户/系统选定一个"强调方向"作为主基线                │
└──────────────────────────────────────────────────────────┘
        │
        ▼
┌─────────────── 演示默认 ────────────────────────────────┐
│  仿真/评审界面默认 = "基线方案"项，后端用 approx_evaluator   │
│  评估库里均衡方案的冻结权重解（路 A）；库空 → 仍回退 ATC     │
└──────────────────────────────────────────────────────────┘
```

## 4. 详细设计

### 4.1 夜间冷搜索（baseline build）

- **输入数据源**：系统已有 `planning_context` / `inst_*` 表存当前车间实例；夜间任务读取当前实例构建 `ShopFloor`。这和你白天用的数据同源，保证"学到的策略在白天仍有效"。
- **求解器**：`HybridNSGA3ALNSOptimizer`，目标键 = `[main_order_tardy_ratio, critical_active_window_utilization, avg_flowtime]`。
- **自增强循环（warm-start 闭环，关键）**：夜间构建与白天优化**共用同一个** `HybridNSGA3ALNSOptimizer._seed_population`（见 §4.5）。因此夜间任务在启动时也会从「当前库里 active 方案」（即**上一夜的结果**）热启动，在 3h 预算内对昨日方案做精细化，而非从头冷探索；产出的更优库再作为次日与次夜的先验。循环为：夜间冷启动（首夜库空）→ 写库 → 次日/次夜以库热启动 → 写更优库 → … 持续抬升。首夜因库为空自动退化为冷启动（图谱引导+11规则+ATC），之后自动转热启动，**完全自驱动、无需人工干预**。
  - **关于「白天最优」**：白天运行的精英当前**不回写库**（库仅由夜间构建写入）；但夜间以 3h 预算从同一库重新推导，天然覆盖白天所得，所以「白天最优」已被间接吸收。若希望白天精英直接喂给次日夜做起点（更紧的闭环），可额外在白天优化结束时把精英方案持久化进库——属可选增强，非必需。
- **算力预算（可配，不写死）**：调度时长由 `time_limit_s`（总秒数）+ `generations`（代数上限）+ `coarse_time_ratio`（coarse 广搜占比）三者共同决定。底层 `HybridConfig` 与 API 请求体 `HybridOptimizeReq` 的默认值都是 `time_limit_s=90`、`generations=12`（这是**交互式优化**的默认值，与本夜间的默认互不影响）。
  - **夜间专用默认（已定）**：`time_limit_s = 10800s（= 3 小时）` 是**真正的预算闸**；`generations = 1440` 只是**上限帽**。二者不是等比换算关系——代数与时间不是同一个旋钮，`generations` 只在"提前达到代数上限"时才起作用。因此不必对它做伪精确推导，**设一个足够大、几乎不会成为瓶颈的值即可**（1440 已远超 3h 在大实例上实际能跑完的代数；跑 4h 也无需改动，时间闸自动放宽）。收敛主要由 `time_limit_s` 与下方 stagnation 早停共同决定。
  - **夜间专用早停（已定）**：`stagnation_generations = 5`（白天默认 3）。夜间不惜时间求质量，连续 3 代前沿池无新解就停偏早（大实例多目标搜索常见"卡几代后再突破"）；放宽到 5 让它别过早放弃、把时间预算用足，代价仅是可能多跑几代无效搜索（夜间无所谓）。总闸仍是 `time_limit_s`。
  - **取值优先级**：**命令行参数 > 端点请求体字段 > 夜间专用默认（10800s / 1440 代 / stagnation 5）> 90s**。夜间任务绝不能依赖那个 90s 默认——必须显式落到夜间专用默认或更高优先级的覆盖值。
  - 以 **coarse 广搜阶段为主**（近似评估，快），建议 `coarse_time_ratio≈0.7~0.75`（Exact 晋升与 ALNS 精修缩短），保证夜间任务不抢占白天资源。
- **触发入口**（见 §8 决策点 2）：独立进程/后台任务，不阻塞白天 API。

### 4.2 方案自适应抽取（数量 = "根据情况，抛出多少就多少"）

这是满足你"方案数量可以是多个、根据情况"要求的核心。抽取算法：

1. **逐目标极端解**：对 3 个目标各取"该目标最优"的解 → 最多 3 个极端方案（若多个目标被同一解同时最优，自然去重）。
2. **膝点解（均衡）**：取超体积贡献最大（或距离理想点最近且被前沿支配最少）的解，代表"三项都不过分差"的均衡策略 —— 这是演示默认与通用主基线的候选。
3. **参考点簇代表**：把 Pareto 前沿按 NSGA-III 的参考点做聚类，每个簇取一个代表解。前沿分布越广，簇越多，方案越多；分布集中则自然少。这一步让"方案数随前沿形状自适应"。
4. **去重**：按权重向量签名（`CandidateParameters.signature()`，现有方法）或目标向量欧氏距离聚类，剔除近似重复。

最终方案数 = 极端(≤3) + 膝点(1) + 簇代表(可变)，去重后通常在 **3–8 个**之间，**完全由当晚前沿形状决定**。每个方案记录一个 `emphasis` 标签（如 `min_tardy` / `max_critical_util` / `balanced` / `cluster_k`）。

### 4.3 质量门禁（防"毒化"，关键）

每个候选方案落库前必须通过。**门禁按方案类型分档**（关键修正）——因为极端解按定义就是"牺牲其余目标去压一个目标"，用统一的"多数目标不劣于 ATC"会把想要的极端解自己拒掉：

- 在**当前快照**上用真实 `Simulator`（或与训练同口径的 `approx_evaluator`，见 §4.6）重评其目标向量；
- 与基线 **ATC** 同快照同评估比较，按方案 `emphasis` 分档：
  - **均衡解（`balanced`）**：要求"**多数目标（≥2/3）不劣于 ATC，且至少一个目标严格更优**"（按方向比较）。
  - **极端解（`min_*` / `max_*`）**：要求"**在其强调的那个目标上显著优于 ATC**（改进幅度 ≥ 阈值，如 10%），且其余目标**不出现灾难性退化**（不劣于 ATC 的某个宽松倍数，如 1.5×）"。
  - **簇代表（`cluster_k`）**：按均衡解档处理。
- 并记录相对 ATC 的改进幅度 `baseline_compare` JSON，作为可信度证据。
- **说明**：若整条前沿都支配 ATC（ATC 是"仅可行"弱规则，常见情形），极端解自然也能过均衡档；分档只是防止"前沿未整体压过 ATC"时把有价值的极端解误杀。

不达标者丢弃，绝不入库。这样即使某晚数据异常/退化，库也不会被污染。

### 4.4 持久化（新建 `baseline_solutions` 表）

风格对齐现有 `data/db.py`（用 `get_db()` 上下文管理器 + `CREATE TABLE IF NOT EXISTS` + WAL 模式）。表结构：

```sql
CREATE TABLE IF NOT EXISTS baseline_solutions (
    id              TEXT PRIMARY KEY,
    emphasis        TEXT NOT NULL,                 -- 强调方向标签
    objective_keys  TEXT NOT NULL,                 -- JSON: [main_order_tardy_ratio, critical_active_window_utilization, avg_flowtime]
    feature_names   TEXT NOT NULL,                 -- JSON: 当时的 FEATURE_NAMES（版本护栏，见 §4.7.2）
    feature_weights TEXT NOT NULL,                 -- JSON: 16 维权重（主基因）
    scale_json      TEXT NOT NULL,                 -- JSON: 夜间归一化尺度（记录+校验，非注入，见 §4.4）
    op_bias         TEXT DEFAULT '{}',             -- JSON: 逐工序偏置
    destroy_weights TEXT DEFAULT '{}',             -- JSON: ALNS 破坏算子权重
    repair_weights  TEXT DEFAULT '{}',             -- JSON: ALNS 修复算子权重
    destroy_fraction REAL DEFAULT 0.3,
    reached_objectives TEXT NOT NULL,              -- JSON: 达到的目标值
    baseline_compare TEXT NOT NULL,                -- JSON: 相对 ATC 的改进幅度（质量门禁证据）
    snapshot_id     TEXT,                          -- 来源快照标识
    snapshot_version INTEGER,                      -- 快照版本号
    batch_id        TEXT NOT NULL,                -- 构建批次标识（用于保留最近2批历史）
    created_at      TEXT NOT NULL,                 -- 构建时间
    status          TEXT NOT NULL DEFAULT 'active',-- active / retired
    source          TEXT DEFAULT 'learned'         -- learned / rule_seeded
);
```

- **尺度处理：记录 + 校验，而非"冻结注入"**（关键修正，对齐代码现实）：加权派工中权重与 time/priority/due 的尺度耦合确实存在，但 `time_scale/due_scale/priority_scale/busy_scale` 是评估器在读入**当前实例**时现算的（`hybrid_nsga3_alns.py:329-332`），当前架构**没有把外部 scale 注入回评估器的入口**（加载走的 `_project_candidate_to_graph_space` 只调权重、不碰 scale）。因此 `scale_json` 的真实作用是**存证 + 校验**：夜间把当时 scale 存入 `scale_json`；白天加载方案时，用它与当前实例现算的 scale 比对，**偏差超阈值（如 20%）则告警/跳过该方案**，否则照常用当前 scale。因为白天/夜间实例同源、只有小调整，这几个粗统计量（平均工时/最大交期/最大优先级/机器数）几乎不变，尺度漂移天然很小，`scale_json` 主要兜住"实例结构大变"的极端情况。（若未来确需真正冻结尺度，再给评估器加"外部 scale 注入口"，属独立改动，本方案不依赖。）
- **`feature_names` 作版本护栏**：记录构建时的 `FEATURE_NAMES` 顺序/集合，加载时校验与当前完全一致，防止特征定义变更导致权重错位（见 §7.2）。

### 4.5 白天热启动（warm start）

- **改动点**：`HybridNSGA3ALNSOptimizer._seed_population`（hybrid_nsga3_alns.py:1413）把库里所有 `status='active'` 的方案（经 `_project_candidate_to_graph_space` 投影到当前图空间，复用现有逻辑）纳入种子。
- **必须解决的截断陷阱**（关键修正）：`_seed_population` 末尾有 `list(unique.values())[: self.config.population_size]`（默认 `population_size=24`）。现有 seeds（图谱引导×2 + 11 内置规则）已接近该上限，**若把基线方案简单追加在列表末尾，一旦总数超 24 就会被静默截掉**，warm start 形同虚设。修正两步走：
  1. **基线方案插在种子列表最前面**（高优先级）——它们是"跑出来的好方案"，应优先保留，排在图谱引导/内置规则之前参与去重与截断；
  2. **相应上调 `population_size`**：随 active 方案数动态放大，`population_size = max(24, 现有种子数 + active 方案数)`（或直接给夜间/热启动一个更大的固定值），确保基线方案**与现有种子都能进种群、互不挤占**。
- **零回归保证**：
  - 库为空 → 行为退化为现有（图谱引导 + 11 规则 + ATC），`population_size` 维持 24；
  - 库存在但加载失败/版本不符 → 跳过，退化到现有行为；
  - 上调 `population_size` 后现有 11 规则、图谱引导种子**一个都不掉**（这正是"插最前 + 扩容"要保证的）。
- **可选主基线**：允许用户/系统选定一个 `emphasis` 作为"主基线"，把对应方案权重以更高 blend 注入；不选则均匀注入所有方案。

### 4.6 演示默认改进

- **评估路径（关键修正）**：不能简单"套 `rule(op, machine, features, shop)` 签名"。查代码：真实 `Simulator._features`（simulator.py:677）的特征字典**不含 4 个图特征**（`predecessor_depth`/`assembly_criticality`/`shared_resource_degree`/`bottleneck_adjacency`），也没有 scale，而 16 维权重恰好依赖这 4 维（它们只在 `approx_eval` 的 `graph_features` 里）。硬套 rule 签名会丢掉 4 个图维度且无归一化，与训练解行为对不上。**因此采用路 A**：
  - 从库里取"均衡方案"（`emphasis='balanced'` 或超体积贡献最大者），演示"基线方案"这一项**由后端直接用 `approx_evaluator`（优化器评估解本来就用它）跑一遍其冻结的 16 维权重解并返回结果**——scale + 图特征齐全，与夜间训练 / 白天热启动**完全同口径、零偏差**，也不碰 `Simulator` 热路径。它是下拉里的一个特殊项，选中时后端走该评估路径，而非规则仿真回调。
  - （备选路 C，若需与内置规则严格同框走真实 `Simulator`：把 `BASELINE_SMART` 定义成**只用真实 Simulator 那 12 个非图特征**的加权 rule，丢弃 4 个图特征——行为与训练解近似不完全一致，但仍强于 ATC，实现最省事。本方案默认走路 A，路 C 仅备查。）
- **前端**：`frontend/app_v2.js:7` 的 `HEURISTIC_RULES` 写死 6 条。改动是**追加**（把"基线方案"加进列表、并设为演示默认首项），不删除现有 6 条。
- **可切换**：演示界面提供"基线方案"下拉，**默认选中均衡方案**，但用户可改选库里其他方案（极端/簇代表）或回退到 ATC / 任一内置规则。下拉项由后端从 `baseline_solutions`（active 批）动态拉取。
- **回退**：库空或蒸馏失败时，演示默认仍是 ATC，前端行为不变。

### 4.7 刷新、失效与回退（"不会带来新问题"的护栏）

1. **构建批次（batch）与历史保留（最近 2 批）**：每次夜间构建分配一个 `batch_id`（时间戳）。落库时：当前所有 active 方案先转 `previous`（上一批），新方案写入 `active`（最新批），更早的 `previous` 转 `retired`（或删除）。即**始终保留"最新批 + 上一批"两份**——即便某晚只跑出单条/异常方案，仍有上一批作为对照与兜底，避免"最新只跑出一条即无历史"的风险。
2. **特征版本失效**（关键护栏）：加载方案时校验 `feature_names` 与当前 `FEATURE_NAMES` 完全一致；**不一致则忽略该方案并回退 ATC**，避免车间模型/特征定义变更导致的静默错误。
3. **复验脚本**（可选但强烈建议）：CI/手动跑一遍，确认库内方案仍稳定优于 ATC；不优于则告警并把演示默认回退到 ATC。
4. **全局兜底**：ATC 永远保留为"最后基线"。任何环节（库空、加载失败、质量门禁未过、夜间任务崩溃）都自动落到 ATC，**无任何副作用**。

## 6. 实施步骤（分阶段，每步可独立验证）

1. **(P0) 目标集确认**：已定用 `[main_order_tardy_ratio, critical_active_window_utilization, avg_flowtime]`（路径 B，零改动）。→ 验证：`validate_objective_selection([...])` 通过；三键方向正确（**两 min**：延误率、流程时间；**一 max**：关键资源活跃窗口利用率）。
2. **(P1) 持久化层**：在 `data/db.py` 加 `baseline_solutions` 表（含 `batch_id`）+ 读写与批次轮转函数（`save_baseline_batch` / `load_active_baselines` / `rotate_baseline_history` 保留最近 2 批）。→ 验证：建表幂等、读写往返一致、轮转后恰好保留最新+上次两批。
3. **(P2) 方案抽取器**：实现 `extract_baseline_solutions(pareto_front, objectives)`（极端+膝点+簇代表+去重）。→ 验证：给定合成前沿，输出方案数合理且含均衡解。
4. **(P3) 质量门禁**：实现 `passes_quality_gate(solution, atc_result, objectives)`，**按 `emphasis` 分档**（均衡解/极端解不同判据，见 §4.3）。→ 验证：退化解被拦截；均衡强解通过；极端解在其强调目标显著优于 ATC 时通过、其余目标灾难性退化时拦截。
5. **(P4) 夜间构建入口（命令行 + 端点）**：
   - 命令行：`tools/build_baseline.py`（读当前实例 → 跑优化 → 抽取 → 门禁 → 入库，分配 `batch_id`）。
   - 后端端点：`POST /baseline/build`（可接 cron / 手动触发）。
   → 验证：两种入口各跑一次，确认产出非空库、ATC 对比合理、`batch_id` 正确写入、历史按"最近 2 批"保留。
6. **(P5) 白天热启动**：改 `_seed_population`——active 方案**插在种子列表最前面**，并把 `population_size` 上调至 `max(24, 现有种子数 + active 方案数)`，避免被末尾截断（见 §4.5）。→ 验证：库空时行为与改前逐字节一致（回归测试）；库非空时所有 active 方案与现有种子都进种群、无一被截掉。
7. **(P6) 演示默认**：后端"基线方案"项走 `approx_evaluator` 评估冻结权重解（路 A，见 §4.6）+ 前端 `HEURISTIC_RULES` 追加默认项。→ 验证：库空时演示仍走 ATC；库存在时默认展示更强结果，且其目标值与夜间训练/白天热启动同口径一致。
8. **(P7) 复验/监控（可选但建议）**：复验脚本（确认库内方案仍稳定优于 ATC，否则告警并演示回退 ATC）+ 批次保留监控。

## 8. 风险与缓解（深度补充）

- **过拟合单一快照**：夜间只用了"当前实例"，权重可能过拟合该分布。**已定：不做扰动变体仿真**——直接用当晚最新排产清单的那一份数据训练即可（用户裁定）。缓解手段仅保留轻量项：质量门禁在**同一快照 + 不同随机种子**上验证鲁棒性；报告 `baseline_compare` 作为可信度证据。白天实例与夜间同源且只有小调整，过拟合风险本就可控。
- **尺度耦合**：`due_date` 等特征含绝对时间量，权重乘它尺度敏感。缓解：`scale_json` 采用"记录 + 校验"（见 §4.4）——存夜间 scale，白天加载时比对当前 scale，偏差超阈值才告警/跳过；同源实例尺度天然接近，正常情形直接用当前 scale。
- **可解释性**：16 维黑箱权重不利讲解。缓解：入库时一并记录"主导特征排名"（权重绝对值 top-k 的特征名），演示时可解释"该方案主要按 X/Y/Z 排序"，也便于日后手工微调。
- **库膨胀/失效堆积**：`retired` 方案长期堆积。缓解：P7 可选加保留份数上限 + 定期清理。
- **与 LLM 进化规则的关系**：LLM 进化也能产规则，但确定性弱、易漂移；本方案权重向量来自优化器、确定可复现，更适合"固化基线"。二者可并存，留验证更优者。

## 9. 一句话总结

> 把"夜间对实际数据做三维多目标大规模优化"产出的**多个极端最优 + 一个均衡最优的派工参数方案**持久化成**基准方案库**；白天和小优化始终从这些方案**热启动**，又快又稳；演示默认用均衡方案。全程以"库可空、可失效、可回退到 ATC"为护栏，不引入任何破坏现有行为的新问题。
