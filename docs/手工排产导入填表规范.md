# 手工排产数据导入填表规范

> 适用场景：在「方案评审」界面导入一套**手工排程**，与算法（Pareto）方案并排对比 KPI / 甘特 / 利用率。
> 本文档给你（或同事）一份可直接照着填的规范。配套 Excel 模板见同目录 `手工排产导入模板.xlsx`。

---

## 0. 一句话结论

手工排产要填的，**只有一张 `schedule` 工作表**（每行 = 一个工序的排程决策：在哪台机、什么时刻开工/完工、占哪些工装/人员）。
但表里引用的所有 `order / task / op / machine / tooling / personnel` 的 **ID 必须和界面「当前已加载实例」完全一致**——因为交期、工时、日历窗、关键机台这些 KPI 参数都来自那套实例主数据，不在你填的表里。

---

## 1. 两套数据，别混

| 数据 | 谁来提供 | 说明 |
|---|---|---|
| **实例主数据**（`inst_*` 表，见 §6） | 系统 /「当前实例」 | 你**不用填**，但引用的 ID 必须与之匹配 |
| **`schedule` 工作表**（本文档重点） | **你（手工排产）** | 每行一个工序的排程决策 |

导入时后端用：`build_schedule_analytics(当前实例, SimResult(schedule=你填的条目))` 算 KPI。
即——**决策在你表里，参数在实例主数据里，两份缺一不可**。

## 2. `schedule` 工作表列定义（19 列，与系统「导出方案」格式完全一致）

| 列名 | 含义 | 类型 | 要求 | 说明 |
|---|---|---|---|---|
| `order_id` | 订单 ID | 文本 | **建议必填** | 评审甘特按订单分面；可由 `op_id → task → order` 推导，但显式给最稳 |
| `order_name` | 订单名称 | 文本 | 可选 | 展示用，可由实例推导 |
| `task_id` | 任务 ID | 文本 | 可选 | 展示用，可由 `op_id` 推导 |
| `op_id` | **工序 ID** | 文本 | **必填** | 必须存在于当前实例工序表 |
| `op_name` | 工序名称 | 文本 | 可选 | 展示用，可由实例推导 |
| `machine_id` | **机台 ID** | 文本 | **必填** | 机台必须存在，且在该工序 `eligible_machine_ids` 内 |
| `machine_name` | 机台名称 | 文本 | 可选 | 展示用，可由实例推导 |
| `tooling_ids` | 占用工装 ID | 文本（逗号分隔） | **条件必填** | 若工序 `required_tooling_types` 非空则必填 |
| `personnel_ids` | 占用人员 ID | 文本（逗号分隔） | **条件必填** | 若工序 `required_personnel_skills` 非空则必填 |
| `start` | 开工时刻 | 数值（小时） | **必填**（与 `start_at` 二选一） | 相对 `plan_start_at` 的**偏移小时数** |
| `end` | 完工时刻 | 数值（小时） | **必填**（与 `end_at` 二选一） | 相对 `plan_start_at` 的偏移小时数，须 `> start` |
| `start_at` | 开工墙钟时间 | 文本（ISO） | 可选（与 `start` 二选一） | 如 `2026-07-23T08:00:00+08:00` |
| `end_at` | 完工墙钟时间 | 文本（ISO） | 可选（与 `end` 二选一） | 同上 |
| `duration` | 时长 | 数值（小时） | 可选 | 一般取 `end − start` |
| `elapsed_duration` | 实际占用时长 | 数值（小时） | 可选 | 工装/人员占用累加用；缺省取 `end − start` |
| `is_main` | 是否主任务 | 0/1 | 可选 | 展示用，可由实例推导 |
| `due_at` | 交期（偏移小时） | 数值 | 可选 | 展示用；KPI 交期取自实例，不取本列 |
| `status` | 状态 | 文本 | 可选 | `completed` / `in_progress` / `scheduled`；纯手工未来计划填 `scheduled` 或留空 |
| `status_label` | 状态说明 | 文本 | 可选 | 展示用 |

## 3. 必填速查

**最小可导入集（每行）：**
```
op_id + machine_id + start + end
```
+ `order_id`（建议显式）
+ `tooling_ids` / `personnel_ids`（当对应工序有约束时必填）

其余列（名称、`*_at`、`duration`、`is_main`、`due_at`、`status` 等）都可省略或由系统推导，仅影响展示丰富度，不影响 KPI 计算。

---

## 4. 时间怎么填（最容易错的地方）

- **推荐只填 `start` / `end`**（相对 `plan_start_at` 的偏移小时）。例：计划起点当日 8:00 开工、12:00 完工 → `start=8, end=12`。
- 系统的「偏移 0 点」= 当前实例的 `plan_start_at`（在 `planning_context` 表）。**锚点弄错会导致整表时间整体偏移**。
- 若只给 ISO 墙钟 `start_at` / `end_at`，后端会按 `plan_start_at` 反算成偏移小时——**两套表示法二选一，不要混用同一行**。
- `end` 必须严格大于 `start`（时长 > 0）。

---

## 5. 导入前的可行性自检清单

后端导入时会校验并返回**逐行违规清单**。自己先核对，能少返工：

1. **工序存在**：每个 `op_id` 都在当前实例工序表中。
2. **机台资格**：`machine_id` 存在，且属于该工序 `eligible_machine_ids`。
3. **前驱 + 周转时间**：对任意前驱工序 `p`，本工序 `start ≥ p.end + p.turnover_time`（`turnover_time` 是一等约束，漏算会让对比口径不一致）。
4. **机台不重叠 / 日历窗**：同一机台的时间区间不相交；且落在班次窗内、避开 `machine_downtime`。
5. **工装 / 人员约束**：有 `required_tooling_types` 的工序必须给 `tooling_ids`，且不与其它占用冲突；`required_personnel_skills` 同理。
6. **完整性**：**覆盖当前实例的全部工序**，每个工序恰好一行。只导部分工序会触发"未排工序"悲观惩罚，使手工方案 KPI 显得更差、对比不公平。

> 注意：KPI 计算本身**不校验可行性**——它只看"每个工序是否都出现在表里"。所以"算得出 KPI"≠"排程可行"，可行性必须由上面这条清单保证。

## 6. 实例主数据字段对照（你引用的 ID 来自这里）

这些表已经在系统里（由「当前实例」提供）。你**不填**，但表里引用的 ID 必须与之对应：

| 原始表 | 提供的实体 / 字段 | 与你的 `schedule` 的关系 |
|---|---|---|
| `inst_orders` | 订单 `order_id`，`due_date`，`release_time`，`priority` | `order_id` 引用；KPI 主订单延误率/装配同步取这里的 `due_date` |
| `inst_tasks` | 任务 `task_id`，`order_id`，`is_main`，`due_date`，`release_time` | `task_id` / `order_id` 引用；KPI 延误/流转取这里的 `due_date`/`release_time` |
| `inst_operations` | 工序 `op_id`，`task_id`，`processing_time`，`turnover_time`，`eligible_machine_ids`，`required_tooling_types`，`required_personnel_skills` | `op_id` 引用；`processing_time` 用于 productive 时间；`eligible_machine_ids` 决定机台资格；`turnover_time` 用于前驱约束 |
| `inst_machines` | 机台 `machine_id`，`shifts` | `machine_id` 引用；`shifts` 决定日历可用窗（净可用利用率） |
| `inst_machine_types` | 机台类型 `is_critical` | 决定 `get_critical_machines()`（瓶颈利用率） |
| `inst_toolings` | 工装 `tooling_id`，`shifts` | `tooling_ids` 引用 |
| `inst_personnel` | 人员 `personnel_id`，`skills`，`shifts` | `personnel_ids` 引用 |
| `machine_downtime` | 机台停机窗 | 机台可用性约束 |
| `planning_context` | `plan_start_at` | 偏移小时的锚点 |

## 8. 常见错误

- ❌ 只导部分工序 → KPI 失真（未排工序被悲观惩罚）。
- ❌ 用错一套实例（不同 `inst_*` 数据）→ 交期/工时/日历口径错乱，和算法方案不可比。
- ❌ 同一行既填 `start/end` 又填 `start_at/end_at` 且不一致 → 锚点混乱。
- ❌ 漏算 `turnover_time` → 前驱约束判错，对比不公平。
- ❌ 机台不在 `eligible_machine_ids` 内 → 导入校验报"机台资格不符"。

---

## 9. 导入后会发生什么

1. 后端解析 `schedule` 条目 → 做 §5 的可行性校验 → 返回逐行违规清单（若有）。
2. 通过校验后，`build_schedule_analytics(当前实例, 条目)` 算全部 KPI（与算法/规则参照方案同管线）。
3. 组装为 `source="manual"` 的参照方案，存入评审库；在评审界面与算法方案并排显示甘特、利用率、KPI 对比、AI 评审。
4. 占用一个评审选择名额（评审最多勾选 4 个方案）。
