LLM4DRD 排产优化算法拆解
01

排产问题，与那台"仿真解码器"

在讲任何优化算法之前，先搞懂：一个"排产方案"到底是什么，又是怎么被算出分数的。

先看清战场

想象一个车间：约 1000 张订单、2 万多道工序。每张订单要先加工一堆子件，再总装、检测；工序之间有先后约束；机器按工艺分组、会停机、会突然故障；每张订单还有交期。

这类问题有个学名：动态柔性装配流水车间调度（DFAFSP）。它难，是因为决策点多到爆炸。

🧩
柔性

一道工序可以在多台同工艺机器上加工——到底派给哪台？

🏗️
装配层次

子件没做完，总装就不能开工——先后顺序不能乱。

⏱️
多目标

要完工快、又要少延期、还要机器别闲着——这些常常互相打架。

关键一招：不直接排时刻表，而是"定规则"

最直觉的想法是：直接给出一张巨大的时刻表——每道工序几点在哪台机器开工。但 2 万道工序的时刻表，可能组合数比宇宙原子还多，而且绝大多数还违反约束（前驱没做完就开工）。

这个系统换了个思路：一个"方案"不是时刻表，而是一套 派工规则。规则决定"每台机器空闲时下一步做哪道工序"，再由一台"解码器"把规则演绎成完整、合法的排程。

💡
乐谱与乐团

把"规则/参数"想成乐谱：它不写死每个音符的物理声波，只写下演奏法则。仿真引擎就是乐团，照着乐谱一路演奏，最终奏出真实的声音——也就是那张合法的排程。优化算法要做的，是不断改写乐谱。

解码器：离散事件仿真

这台解码器是一个 离散事件仿真引擎。它不逐秒空转，而是用一个最小堆存放未来事件，每次跳到最近的事件时刻。整个车间的演化，被压缩成三类事件的循环：

派工，就发生在这一段

下面是仿真主循环里 dispatch 事件的心脏：机器对每个候选工序算出特征、交给规则打分，用一个带平局判据的元组挑出最优。这就是"规则被演绎成决策"的确切瞬间。

CODE · core/simulator.py 
for op_id in candidate_ids:
    features = self._features(op, task, order, machine, shop, now)
    score = self.pdr(op, machine, features, shop)
    tie_break = op.work_remaining
    candidate = (score, -tie_break, op.id, op, toolings, people)
    if best is None or candidate[:3] > best[:3]:
        best = candidate
大白话

遍历这台机器就绪队列里的每一道候选工序。

先算出这道工序此刻的一组"特征"（有多急、剩多少活……下个模块细讲）。

pdr 就是当前那条派工规则——把特征喂进去，得到一个优先级分数。

平局判据：分数一样时，剩余工时更少的优先（这里用负号让"小"排到"前"）。

把分数、平局判据、工序打包成一个元组。

按元组前三项比大小，谁大谁当"目前最优候选"。

循环结束，best 就是这台机器这一刻该开工的工序。

🧭
为什么这台解码器如此关键

后面每一个优化算法——规则枚举、NSGA-III、ALNS——本质都在做同一件事：造出不同的规则/参数，丢给这台解码器跑一遍，看目标向量好不好。解码器是所有算法共用的"评分裁判"。

跑完一次，得到什么？

解码器吐出的不是一个数字，而是一个 目标向量——一整排 KPI。正是这排数字，让不同方案能被比较、被优化。

🏁

Makespan

从开工到最后一道工序完成的总跨度。越小越好。

⏰

总延迟

所有任务超过交期的时间之和。越小越好。

⭐

主订单延误率

重点订单里踩线的比例。越小越好。

📈

机器利用率

机器有多少时间在真正干活。越大越好。

02

规则固定派工：把"决策"变成"旋钮"

优化算法搜的不是排程，而是这套派工规则的参数。先看懂规则本身。

一台机器的分诊台

机器空闲的那一刻，面前排着一堆已就绪的工序。谁先上？这和急诊分诊一模一样：护士不是先到先看，而是给每个病人打一个"危急分"，分高的先进抢救室。派工规则就是那张打分表。

系统内置 11 条这样的规则。每条都是一个纯函数：输入工序的特征，输出一个分数，越大越优先。

CODE · core/rules.py 
def atc_rule(op, machine, f, shop):
    processing = f["processing_time"]
    if processing <= 0:
        return 1000.0
    slack = max(f["slack"], 0.0)
    return (1.0 / processing) * math.exp(-slack / (2.0 * processing + 0.01))
大白话 · ATC 规则

ATC = 表观延迟成本。它同时看两件事：这活多快能干完、还有多少余量。

加工时间为 0 的异常工序直接给个极大分，让它立刻走人。

松弛取非负部分：真正紧急（松弛为负）的会被下一行的指数放大。

分数 = (1÷加工时间) × e 的 (−松弛/加工时间) 次方。

翻译：活越短、越紧急，分越高——短活先清、快烧眉毛的插队。

🩺
11 条规则 = 11 种分诊哲学

EDD 只看交期（谁快到期谁先），SPT 只看谁活最短，KIT_AWARE 偏爱配套齐全的，BOTTLENECK 让瓶颈工序优先……每条都是一种"该在乎什么"的立场。

从"选一条规则"到"调一台调音台"

只有 11 条固定规则太死板。真正的杀招是：把打分公式参数化成 16 个特征的加权和。每个特征配一个权重，就像调音台上的 16 个推子——推子的组合，定义出无穷多种派工风格。

🎚️
这一步是全局关键

一旦规则变成"16 个权重"，"设计一条好规则"就变成了"找一组好权重"——而找一组好数字，正是优化算法最擅长的事。模块 4、5、6 优化的，全是这组权重。

CODE · optimization/approx_eval.py 
score_components = {
    "urgency": features["urgency"] / self.time_scale,
    "slack": -features["slack"] / self.time_scale,
    "processing_time": -features["processing_time"] / self.time_scale,
    # ... 共 16 个特征分量
}
score = sum(candidate.feature_weights.get(name, 0.0)
            * score_components.get(name, 0.0) for name in FEATURE_NAMES)
score += candidate.op_bias.get(op.id, 0.0)
大白话 · 权重 × 特征 = 分数

先把每个特征归一化成一个"分量"（除以尺度，让不同量纲可比）。

紧急度正向加分；松弛越小越加分（前面带负号）。

加工时间前带负号 → 活越短越加分。

总共 16 个这样的分量，覆盖交期、装配关键度、瓶颈邻接等。

核心一行：分数 = 候选的 16 个权重 与 16 个特征分量的点积。

再叠加 op_bias：针对个别关键工序的微调偏置。

一句话：一组权重（candidate）= 一条派工规则。

"候选解"到底是什么

于是优化算法眼里的一个"解"，就是下面这个 CandidateParameters：一小包数字，而不是一张排程。排程要等它被解码器跑过才生成。

feature_weights16 维特征权重——决定派工打分的"性格"，是被优化的主战场
destroy_weights对各"破坏算子"的偏好（模块 5 的 ALNS 会用到）
repair_weights对各"修复算子"的偏好
op_bias对个别关键工序的定点微调偏置
seed_rule_name混入一条内置规则作"底色"，稳住搜索起点

03

Pareto 前沿：没有唯一最优，只有权衡边界

当"快"和"准时"互相打架时，"最优解"这个词本身需要重新定义。

两个目标，一场拉锯

你想要 Makespan 最小（尽快清完），又想要总延迟最小（别让谁超期）。问题是：为了压低总延迟，你会插队赶那些快到期的活，结果机器切换频繁、Makespan 反而变长。它们常常此消彼长。

这时"哪个方案最好"没有唯一答案。数学上用一个精确概念来判断优劣：支配。

✅
A 支配 B

A 每个目标都 ≥ B，且至少一个严格更好 → B 是纯粹的输，淘汰。

⚖️
A、B 互不支配

A 完工更快但延期更多，B 反过来 → 各有千秋，都得留着。

支配，写成代码就六行

CODE · optimization/solution_model.py 
def dominates(self, other, specs):
    any_better = False
    for spec in specs:
        left  = self.objectives[spec.key]
        right = other.objectives[spec.key]
        if spec.direction == "min":
            if left > right: return False
            if left < right: any_better = True
    return any_better
大白话

逐个目标比较 self（我）和 other（对方）。

读出双方在这个目标上的数值。

对"越小越好"的目标：只要我在任何一项上更差（更大），立刻判负——我没资格支配它。

只要我在某一项上严格更好（更小），就记下"至少赢了一项"。

全程没输、且至少赢一项 → 我支配它。这就是支配的全部定义。

注意 spec.direction：利用率这类目标是"越大越好"，方向相反。

前沿：山脊线，不是山顶

把所有"没被任何方案支配"的解收集起来，就得到 Pareto 前沿。它不是一个点，而是一条边界。

🏔️
登山者的山脊线

前沿像一条山脊线：线上每一点，往任何方向再走都得下坡（牺牲某个目标）。优化的目标不是"登顶单一最高峰"，而是把这条脊线找全、找准——让决策者能在"快"和"准时"之间自由挑一个平衡点。

系统用一个 ParetoArchive 实时维护这条脊线。每来一个新解，就做一次"擂台裁决"：

CODE · optimization/archive.py 
def consider(self, solution):
    if any(e.dominates(solution, self.specs) for e in self._solutions):
        return False
    survivors = [e for e in self._solutions
                 if not solution.dominates(e, self.specs)]
    survivors.append(solution)
    self._solutions = survivors
    return True
大白话 · 擂台裁决

新解上擂台。如果存档里已有任何一个解支配它——它是纯输，直接拒收。

否则，反过来把存档里被这个新解支配的旧解全部踢下擂台。

把幸存的旧解 + 这个新解，组成新的前沿。

返回 True 表示"入选前沿"。（真实代码还会先按排程/目标去重。）

这样存档永远只装着互不支配的解——一条随搜索不断被刷新的脊线。

前沿好不好，怎么量？超体积

两条前沿摆在一起，哪条更优？看谁"罩住"的目标空间更大。这个指标叫 超体积。系统用蒙特卡洛采样来近似它：

1
归一化

把所有目标缩放到 [0,1]，消除量纲差异。

2
随机撒点

在参考立方体里撒 1500 个随机探针。

3
数比例

统计有多少探针被前沿"罩住"，比例 × 立方体体积 ≈ 超体积。

🎯
为什么用采样近似

精确算超体积在高维目标下代价指数级。撒点估比例——用可控的随机误差换来近乎线性的速度，是工程里典型的"够用就好"取舍。

检验一下理解

两目标都"越小越好"。方案 A =(Makespan 100, 总延迟 50)，方案 B =(Makespan 120, 总延迟 40)。谁支配谁？

既然前沿上的解互不支配，系统最终交给排产决策者的应该是什么？

04

NSGA-III：用"参考点"驾驭多目标进化

遗传算法负责进化候选权重，参考点负责让前沿铺得又满又匀。

进化论，用在一堆权重上

NSGA 是一类遗传算法。这里的"个体"就是模块 2 那包权重（CandidateParameters）。每一代：交叉变异产生后代 → 仿真解码打分 → 择优存活。难点全在最后一步——怎么择优。

🖼️
策展人的难题

你是画展策展人，展厅（目标空间）很大，展位（种群名额）有限，投稿（候选解）成百上千。若只挑"最惊艳"的，它们可能全是同一风格、挤在展厅一角。高明的策展人会把作品均匀铺满每个区域——这正是 NSGA-III 相比老版 NSGA-II 的升级点。

第一刀：非支配排序，分层

先把整个种群按支配关系剥洋葱：谁也不被支配的是第 0 层（当前最强前沿），剥掉后剩下里谁也不被支配的是第 1 层……存活名额从第 0 层往下按层填。

CODE · optimization/nsga3_core.py 
selected_indices, last_front = [], []
for front in fronts:
    if len(selected_indices) + len(front) <= population_size:
        selected_indices.extend(front)
    else:
        last_front = front  # 装不下的临界层
        break
大白话

从最强的第 0 层开始，一层层往存活名单里塞。

只要整层都塞得下，就整层收编——它们都是当前的非支配精英。

直到遇到一层"塞一半就满了"——这层叫临界层（last_front）。

临界层里该留谁、该舍谁？这才是 NSGA-III 真正的用武之地。

第二刀：参考点 + niche，保多样性

面对临界层，老版 NSGA-II 用"拥挤度距离"（谁周围空谁留）。但目标一多，拥挤度就失灵。NSGA-III 改用 参考点——像在展厅地面预先划好均匀的网格，每个候选解归属到最近的一个格子。

CODE · nsga3_core.py · _niche_select 
while len(selected) < population_size and remaining:
    niche_min = min(selected_by_ref[i] for i in ...)
    niche_indices = [i for i in ... if selected_by_ref[i] == niche_min]
    # 从『最空』的参考点里，挑一个离它最近的候选补进来
    selected.append(chosen_candidate)
    selected_by_ref[chosen_reference] += 1
大白话 · niche 保育

还没填满就继续：先找出目前关联解最少的那些参考点（最空的格子）。

niche 保育：优先照顾这些冷清方向，而不是继续往热闹的地方堆。

从最空的格子里，挑一个离该参考方向最近的候选，补进存活名单。

这个格子计数 +1，下一轮它就没那么"空"了，注意力转向别处。

结果：存活者被均匀地摊在整条前沿上，而不是挤成一坨。

🆚
为什么是 NSGA-III 而不是 II

目标数 ≥ 3 时，几乎所有解都互不支配、"拥挤度"区分不开谁疏谁密，NSGA-II 的多样性机制会崩。参考点用固定的几何骨架强行撑开前沿，正好对症这个车间的多目标场景。

05

ALNS：破坏、修复、再破坏

拿一个好解反复"拆一块、重拼一块"，专治遗传算法磨不动的局部最优。

卡在半山腰怎么办

遗传算法能找到一个不错的解，但常常卡在局部最优——周围小改都变差，可它并不是最好的。ALNS 的破局思路很暴力：不小改，而是拆掉一大片重来。

🧩
拼图返工

一幅快拼好的拼图，某个区域别扭。你不会一块块微调，而是整片拆下来重拼——这就是"大邻域"。而哪种拆法、哪种拼法屡屡奏效，你会记住、下次更常用它——这就是"自适应"。

💥

破坏算子

拆掉一片派工决策，比如"延误订单相关的"、"瓶颈机器上的"、"共享工装冲突的"。

🔧

修复算子

把拆掉的部分重排，比如"按交期"、"优先主订单"、"给瓶颈削峰"。

"自适应"：让好算子出头

系统有 6 种破坏 + 5 种修复算子。每次用轮盘赌按权重挑一对。用完后，根据战绩即时调整它的权重：

CODE · optimization/alns_core.py 
def _update_operator(self, state, accepted, improved):
    state.uses += 1
    if improved:
        state.weight = min(6.0, state.weight * 1.12 + 0.08)
    elif accepted:
        state.weight = min(6.0, state.weight * 1.04 + 0.03)
    else:
        state.weight = max(0.2, state.weight * 0.94)
大白话 · 战功考核

每个算子记一笔"出场次数"。

若这次带来了改进 → 权重大涨（×1.12），下次更可能被选中。

若没改进但结果被接受了 → 权重小涨（×1.04），有功劳也有苦劳。

若被拒 → 权重打折（×0.94），但保底 0.2，永不彻底出局。

于是好用的算子自然浮上来，废柴沉下去——却都保留一线生机去探索。

关键一步：故意接受"更差"的解

ALNS 借用了模拟退火的接受准则——有时明知变差也接受，为的是翻过山头去够更高的山。

CODE · alns_core.py · refine 
delta = candidate_score - current_score
if delta <= 0 or self.rng.random() < math.exp(-delta / max(0.05, temperature)):
    accepted = True
    improved = delta < 0
# ... 每轮末尾
temperature *= 0.92
大白话 · 退火接受

delta = 新解得分 − 当前得分（这里越小越好，负的 delta 表示变好）。

变好（delta≤0）就无条件接受。

变差也别急着拒：按 e^(−delta/温度) 的概率接受它。温度越高，越敢接受烂解。

delta 只有严格变好时才算 improved（影响算子考核）。

每轮结束温度 ×0.92 慢慢降——前期大胆探索，后期收敛精修。

混合框架与精确求解：广搜、精修、标尺

NSGA-III 铺广度、ALNS 挖深度、CP-SAT 定标准——三者如何在一条流水线上分工。

找矿队的分工

把整套优化想成一支找矿队。矿区（参数空间）辽阔，精细勘探（完整仿真）昂贵。聪明的做法是分三层力度：

🛰️
卫星遥感（广搜）

NSGA-III + 近似评估，快速扫过整片矿区，圈出富矿带。

⛏️
钻探队（精修）

对晋级的富矿带用完整仿真 + ALNS 精细勘探。

🔬
实验室化验（标尺）

CP-SAT 精确求解，为质量定一个可信基准。

两个"评分裁判"：近似 vs 精确

混合框架的省钱关键，是备了两台评分器——广搜阶段用便宜的近似评估器初筛海量候选，只有晋级的少数精英才享受昂贵的完整仿真。

⚡

近似评估

简化排程，快一个量级。广搜阶段成百上千次调用都靠它。

🎯

精确仿真

模块 1 那台完整解码器，含资源日历、故障。只给晋级精英用。

💰
计算预算的艺术

这就是"广撒网用便宜工具、收网时才上贵工具"。把有限的仿真算力,精准砸在最可能是好解的候选上——工程里典型的预算分配思维。

两阶段流水线

大白话 · 广搜 → 晋级

阶段一（广搜）：一代代进化候选权重。

每代交叉变异出一批后代候选。

用近似评估器并行快速给它们打分（便宜、能海量跑）。

父代 + 后代合并。

用 NSGA-III 的 select_survivors（模块 4）择优——参考点保持前沿多样。

阶段二（晋级）：从广搜积累的候选池里，挑出最值得深挖的精英。

这些精英随后进入精确仿真 + ALNS（模块 5）局部精修，产出最终前沿。

标尺：CP-SAT 精确求解

启发式算法再聪明也不保证"最优"。所以系统请出 CP-SAT 精确求解器：把排产问题原原本本翻译成数学模型，求真正的最优（或给出可信的边界），当作衡量启发式解好坏的标尺。

CODE · optimization/exact.py · solve() 
for op in decision_ops:
    duration = max(1, int(round(op.work_remaining * scale)))
    start = model.NewIntVar(0, horizon, f"s_{op.id}")
    end   = model.NewIntVar(0, horizon, f"e_{op.id}")
    interval = model.NewIntervalVar(start, duration, end, f"i_{op.id}")
# 同一台机器上的所有工序区间，两两不许重叠
model.AddNoOverlap(machine_intervals[resource_id])
大白话 · 建模

为每道待决策工序建三个变量：开工时刻、完工时刻。

加工时长用 scale（×60）把小时换成整数，CP-SAT 只吃整数。

start / end 被限制在 [0, horizon] 这个时间范围内。

区间变量把开工-时长-完工绑成一个对象。

核心约束：同一台机器上所有区间 AddNoOverlap——一台机同一时刻只能干一道工序。

（多台可选机器时还会用布尔选择器 + AddExactlyOne 选一台。）

🐢
精确 = 慢，所以只当标尺

CP-SAT 在两万工序上会爆炸，通常只在小规模实例上求最优，用来验证"启发式解离最优有多近"。日常大实例排产，还是靠混合框架的启发式产出。
