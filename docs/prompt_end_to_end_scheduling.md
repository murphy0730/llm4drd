## 任务一：审视四个步骤是否合理
先读 demo.py 及各步骤实现，梳理出每步的输入/输出与衔接关系。再从以下维度审视「四阶段划分」：
1. 完整性：是否覆盖「数据→建模→求解→结果」全链路，有无关键环节缺失？
2. 顺序性：步骤间依赖与先后次序是否正确，是否存在必须先做后做才能衔接？
3. 冗余/断点：是否存在重复、可合并步骤，或步骤间衔接断裂（上游产物下游用不上）？
4. 端到端一致性：每步输出是否正好是下一步输入？
5. 执行效率（新增维度）：每步在目标数据规模下的时间复杂度是否可接受？是否存在全扫描 /
   重复重建 / 同步阻塞 / 全量渲染等已知低效模式？
给出结论：当前四步骤是否合理，不合理处指出具体调整。

## 任务三：效率瓶颈定位与优化（重要）
若某一步明显偏慢（如仿真在大数据规模下超过数分钟仍未完成，或前端渲染卡死），按以下思路
定位并优化：
- 仿真/求解慢：常见瓶颈在派工循环——对空闲机器遍历「该工艺类型全部就绪工序」再过滤
  eligible（core/simulator.py 的 dispatch 路径），以及每候选重建 features 字典、每次
  op_done 遍历全部机器找 IDLE（_trigger_idle_dispatches）。优先用反向索引
  （machine→可派工序、type→机器）替代全扫描，并对不变特征做缓存。
- 前端卡死：检查是否一次性渲染全部工序到甘特图/图谱（上万条 DOM 会崩溃），以及后端
  /api/simulate 是否同步阻塞、返回超大 JSON。优先改为后台任务 + 进度轮询、分页/抽样/
  虚拟滚动渲染、必要时服务端下采样。
- 优化原则：最小必要改动；改动前先说明文件/逻辑/验证方式；改后复测并对比耗时证明提速；
  涉及核心引擎（core/、optimization/）算法本体的改动需格外谨慎，优先索引/缓存/边界处理。
