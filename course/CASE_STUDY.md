# 案例研究：用 `codebase-to-course` 把排产优化引擎变成一门交互式课程

> 一次真实的实践记录：把 `codebase-to-course` 技能安装进 Claude Code，对准 LLM4DRD 排产平台的 `optimization/` 内核，产出一份可离线运行的单页交互课程。

---

## 一、概要（Summary）

| 项目 | 内容 |
|---|---|
| **使用工具** | Claude Code + `codebase-to-course` 技能 |
| **技能来源** | <https://github.com/zarazhangrui/codebase-to-course>（作者 Zara） |
| **目标代码库** | LLM4DRD 智能调度平台（Python / FastAPI，论文算法实现 + 全栈 Demo） |
| **教学主题** | 排产优化内核的五大核心算法 |
| **产物** | 一个 89 KB、自包含、可离线打开的交互式 HTML 课程（`course/index.html`） |
| **结构** | 6 个模块 · 11 段代码↔中文逐行对照 · 2 个数据流动画 · 1 个群聊动画 · 6 组应用型测验 |

一句话：**把一份需要算法背景才能读懂的排产优化源码，转化成一门"边看代码边看大白话、还能自测"的可视化课程。**

---

## 二、背景（Background）

### 2.1 这个技能是做什么的

`codebase-to-course` 是一个 Claude Code 技能（Skill），它的定位很明确：

> **把任意代码库变成一门漂亮、可交互的单页 HTML 课程，向"非科班"的人讲清楚这套代码是怎么运作的。**

它的设计哲学有几条很鲜明的主张：

- **先构建、后理解**：颠倒传统 CS 教育顺序——不是先啃几年概念再动手，而是"你已经用过/建过这东西了，现在来看它引擎盖下发生了什么"。
- **展示而非灌输**：每屏至少 50% 是可视化（图示、动画、卡片、代码对照），文字块不超过 2–3 句。
- **代码原样呈现**：所有代码片段都是从真实代码库**逐字摘取**（标注文件路径），而不是简化改写——这样学习者打开真实文件能看到一模一样的代码，建立信任。
- **测验考应用而非记忆**：不问"API 是什么缩写"，而问"你想加一个功能，该改哪些文件"。
- **每个概念配一个专属隐喻**，且绝不重复使用（严禁到处套"餐厅"这种烂大街比喻）。

### 2.2 技能的工程设计

它的产物不是一个大文件，而是一个**目录**：预置的 `styles.css` / `main.js`（逐字复制，AI 从不重新生成样板代码）+ 定制的外壳 `_base.html` + 每个模块独立的 HTML 片段 + 一个 `build.sh` 把它们拼成 `index.html`。这个设计的好处：

1. AI 只需专注写"内容 HTML"，不浪费 token 重造 CSS/JS；
2. 每个模块独立撰写，输出体积小、质量高；
3. 拼装后的 `index.html` 离线可用、零配置。

### 2.3 被讲解的对象：LLM4DRD

LLM4DRD 是论文 *"LLM-Assisted Automatic Dispatching Rule Design for Dynamic Flexible Assembly Flow Shop Scheduling"* 的完整实现，面向**动态柔性装配流水车间调度（DFAFSP）**——约 1000 张订单、2 万多道工序，带子件装配层次、交期、机器停机与故障。其中 `optimization/` 目录是整个平台最硬核的部分，也是本课程聚焦的对象。

## 三、问题描述（Problem）

### 3.1 核心痛点

`optimization/` 目录里躺着一批教科书级但门槛很高的算法实现：

```
optimization/
├── nsga3_core.py          NSGA-III：参考点、归一化、niche 保育
├── alns_core.py           ALNS：破坏-修复算子、自适应权重、模拟退火
├── exact.py               CP-SAT 精确求解（OR-Tools）
├── pareto.py / archive.py Pareto 支配、非支配存档、超体积
├── solution_model.py      候选参数（规则固定派工的参数化）
└── hybrid_nsga3_alns.py   两阶段混合框架（2186 行）
```

**问题在于**：这些文件对没有多目标优化 / 元启发式背景的人几乎是天书。团队里的产品、业务、新人，明明每天在用这套排产系统，却说不清"NSGA-III 和 NSGA-II 差在哪""ALNS 为什么要故意接受更差的解""为什么留着一个跑不动大实例的 CP-SAT"。**代码与理解之间存在一道鸿沟。**

### 3.2 实施过程中的关键一步：澄清范围

初次分析代码库时，仓库里同时存在 `pareto.py:NSGA2Optimizer` 和 `ai/evolution.py`（LLM 双专家进化）等模块，很容易被误当作核心。**使用者中途明确澄清了教学范围**：

> "我的核心是 NSGA-III、ALNS、精确求解、规则固定派工、Pareto 前沿最优……没有使用 NSGA-II 和 LLM 双专家进化。"

以及：

> "我希望你主要体现实现逻辑和核心算法讲解。"

这一步至关重要——它把课程从"泛泛介绍平台功能"收敛到"讲透五个指定算法的实现机制"，并明确排除了不相关的模块。**案例启示：面对多分支、易混淆的代码库，先对齐"要讲什么、不讲什么"，比急着产出更重要。**

### 3.3 目标读者与教学目标

| 读者画像 | 教学目标 |
|---|---|
| 用过/维护这套排产系统，但缺算法背景 | 能说清每个算法"在解决什么、怎么解决" |
| 想读懂 `optimization/` 源码的新人 | 把每个文件对应到一个清晰的算法角色 |
| 需要和算法工程师对话的产品/业务 | 掌握精确的术语（支配、参考点、退火、超体积……） |

## 四、解决方案（Approach）

整个流程遵循技能内置的方法论：**分析 → curriculum 设计 → 逐模块构建 → 组装**。

### 4.1 深度代码分析

不是泛读，而是**结构化提取真实代码片段**（带行号）：先用 `grep` 抓取各文件的类/函数骨架，再精读核心算法段落——NSGA-III 的 `select_survivors`、ALNS 的 `_update_operator` 与退火接受、CP-SAT 的区间变量建模、`_priority_score` 里"16 维权重点积"的桥接等。这些片段后续原样进入课程的代码对照块。

### 4.2 课程大纲设计（覆盖五大核心概念）

| # | 模块 | 讲透的算法机制 | 隐喻 |
|---|---|---|---|
| 1 | 排产问题与仿真解码器 | 离散事件仿真事件循环；"解=派工参数→仿真解码"的评估底座 | 乐谱与乐团 |
| 2 | **规则固定派工** | 16 维特征权重 × 特征分量点积 → 派工打分 | 分诊台 / 调音台 |
| 3 | **Pareto 前沿** | 支配关系、非支配存档、超体积估计 | 登山者的山脊线 |
| 4 | **NSGA-III** | 非支配分层 + 参考点关联 + niche 保多样性 | 画展策展人 |
| 5 | **ALNS** | 破坏-修复 + 自适应权重(×1.12/×1.04/×0.94) + 模拟退火 | 拼图返工 |
| 6 | 混合框架 & **精确求解** | 两阶段广搜/精修；CP-SAT 建模当标尺 | 找矿队三层力度 |

每个隐喻只用一次，且都为该概念"量身定制"。

### 4.3 逐模块构建

每个模块独立成一个 HTML 片段，统一包含：代码↔中文逐行对照、至少一个交互元素、"啊哈"要点框、专属隐喻、应用型测验、术语悬浮提示。最后 `bash build.sh` 拼装成 `index.html`。

### 5.1 交付产物

```

├── styles.css      ← 设计系统（原样复制，未改动）
├── main.js         ← 交互引擎（原样复制，未改动）

```

### 5.2 客观指标

| 指标 | 数值 | 校验方式 |
|---|---|---|
| 模块数 | 6 | `grep -c 'class="module"'` |
| 代码↔中文对照块 | 11 | 全部逐字摘自真实源码并标注文件路径 |
| 数据流动画 | 2 | 仿真三类事件循环 / NSGA-III 一代进化 |
| 群聊动画 | 1 | 把 ALNS 一轮迭代演成 4 角色对话 |
| 应用型测验 | 6 组（每模块 1 组，2 题） | 全部为"应用/推理"型，不考死记 |
| `data-steps` 动画 JSON | 2/2 有效 | Python `json.loads` 逐个解析通过 |
| 导航一致性 | 6 导航点 ↔ module-1…6 | 脚本校验一一对应 |

### 5.3 质量特征

- **代码可信**：11 段代码全部来自 `optimization/simulator/rules` 等真实文件、标注路径，学习者可对照原文件核验。
- **精准聚焦**：严格覆盖指定的五大算法（规则固定派工、Pareto 前沿、NSGA-III、ALNS、精确求解），并明确**排除** NSGA-II 与 LLM 双专家进化。
- **测验测应用**：例如"想让长加工工序优先，该把哪个权重调成负值""ALNS 为何按概率接受更差的解""为什么保留跑不动大实例的 CP-SAT"——考的是判断与推理，不是背诵。
- **术语沉淀**：支配、niche、参考点、超体积、模拟退火、CP-SAT、区间变量等算法术语均配中文悬浮定义。

| | 之前 | 之后 |

## 六、结论与启示（Takeaways）

1. **代码库即课程素材**。当一份代码承载了值得传播的知识（尤其是算法密集型项目），`codebase-to-course` 能把"读源码"这件高门槛的事，转化成"看一门可交互课程"。

2. **"逐字引用真实代码"是信任的关键**。课程没有为了好看而简化代码，而是挑选天然精炼的片段原样呈现——这让课程和代码库始终可对照、不脱节。

3. **范围对齐先于产出**。面对分支多、易混淆的代码库，明确"讲哪些、不讲哪些"能极大提升成果的精准度。本案例中一次中途澄清，直接决定了课程的正确走向。

4. **算法讲解 = 机制 + 隐喻 + 可自测**。把"实现逻辑"讲清楚，靠的不是堆术语，而是"专属隐喻建立直觉 + 逐行对照落到代码 + 应用型测验检验理解"三者合一。
