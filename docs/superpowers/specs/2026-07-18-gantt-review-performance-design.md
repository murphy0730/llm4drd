# 甘特图与方案评审交互、读取性能优化设计

**日期：** 2026-07-18

**状态：** 已完成方案评审，待书面规格确认

**范围：** 甘特图默认窗口与订单筛选、方案对比表头、机器分类利用率表、评审页方案勾选与订单切换性能

## 1. 目标

本次优化同时解决五类问题：

1. 甘特图打开后默认展示约 4 天排产，而不是把完整排程周期一次压缩到可视区域。
2. 甘特图订单筛选支持订单号和订单名称的模糊搜索，订单量大时仍可快速定位。
3. 方案对比表第一列表头删除“选”字，但保留复选框及无障碍语义。
4. 机器分类利用率表收窄机器类型列、完整展示方案表头，方案数据格只显示百分比。
5. 消除勾选方案、切换订单时的同步规则重算、重复请求、全量排程重复扫描和整页重复渲染。

本次不修改排产算法、目标函数、利用率计算口径和方案选择上限。

## 2. 当前实现与根因

### 2.1 甘特图默认窗口

`buildGanttData` 当前把可见条目的最早开始到最晚结束作为 `window`，`mountGantts` 再把整个窗口传给 vis Timeline。排程跨度较大时，初始视图会展示完整周期，导致条块过度压缩。

### 2.2 订单下拉无法搜索

普通甘特图和评审联动甘特图都使用原生 `<select>` 渲染订单选项。原生选择框不能按输入内容过滤，订单数量多时只能滚动查找；同时，当前方案排程接口每次都会重新生成完整订单 facet。

### 2.3 利用率表宽度与内容

机器类型列固定为 200px，方案列固定为 120px，方案名称又被限制为 90px、单行省略。方案数据格同时展示百分比、使用机器数和色条，压缩了后部方案列的空间。

### 2.4 评审联动慢

当前勾选方案会触发以下链路：

1. `renderCurrentPage()` 重建整个评审页。
2. `ensureTypeUtilization()` 启动利用率请求，并在请求前后再次重建整页。
3. `ensureReviewGanttData()` 按勾选方案数量发送 N 个排程请求，并在请求前后再次重建整页。
4. 整页 `innerHTML` 替换使旧甘特 DOM 脱离，随后销毁并重新创建 vis Timeline。

切换订单时，同样会按方案数量发送 N 个请求。每个请求都遍历该方案的完整排程，重新构造订单 facet，再扫描一次筛选目标订单。

### 2.5 启发式规则缓存被绕过

`_resolve_export_solution` 遇到 `RULE:*` 时，在检查任务参考方案和规则缓存之前直接调用 `_rule_reference_solution`。因此：

- 加载启发式方案甘特可能重新执行完整规则仿真；
- 加载机器类型利用率可能再次执行相同规则仿真；
- 两类请求并发时，同一规则可能被重复计算；
- 相关端点为 `async def`，其中的同步 CPU 计算会占用事件循环。

这是大实例下秒级、分钟级等待的首要根因。

## 3. 方案选择

### 方案 A：仅前端调整

只修改甘特窗口、订单搜索组件和表格样式。

- 优点：改动小、视觉问题可快速解决。
- 缺点：无法消除规则重算、N 次请求和全排程重复扫描，性能问题仍会存在。

### 方案 B：前端交互调整 + 评审读取链路专项优化

增加规则缓存优先解析、方案排程读取索引、批量评审接口、前端请求取消和局部刷新。

- 优点：直接处理已确认根因；不改变排产算法；可以渐进落地并独立测试。
- 缺点：需要同时修改前端状态管理和后端评审读取接口。

### 方案 C：持久化评审物化视图

优化结束后，把每个方案的订单索引、利用率结果和甘特读取数据持久化到数据库。

- 优点：服务重启后仍可直接读取，适合多用户和超大实例。
- 缺点：需要增加持久化结构、迁移和版本一致性管理，当前需求下复杂度偏高。

**决定：采用方案 B。** 服务端先使用进程内惰性索引；服务重启后从已持久化的完整方案排程重新建立索引，不新增数据库表。

## 4. 交互设计

### 4.1 甘特图默认 4 天窗口

甘特数据同时维护两个范围：

- `fullWindow`：当前数据的完整时间范围，用于拖动、缩放和边界约束。
- `initialWindow`：首次展示范围，目标跨度为 96 小时。

窗口计算规则：

1. 完整排程跨度不超过 96 小时时，展示完整排程。
2. 存在排程进度线时，以进度线为锚点，优先展示约 12 小时历史和 84 小时未来。
3. 进度线靠近排程起点或终点时平移窗口，尽量保持完整 96 小时。
4. 没有有效进度线时，从当前可见排程的最早开始时间展示 96 小时。
5. 切换方案或订单时应用新的 `initialWindow`；机器筛选、分页等同一数据集内操作不强制重置用户手动缩放。

vis Timeline 仍保留水平拖动、滚轮缩放和完整数据集，96 小时只控制初始可视范围，不裁剪排程条目。

### 4.2 可搜索订单组合框

用可搜索组合框替换原生订单 `<select>`，普通甘特图和评审联动甘特图复用同一个组件。

交互规则：

- 点击或聚焦后打开候选列表。
- 支持按订单号、订单名称做不区分大小写的包含匹配。
- 排序优先级为：订单号精确匹配、订单号前缀匹配、订单号包含匹配、订单名称包含匹配。
- 输入防抖 200ms；每次最多展示 50 条结果。
- 支持方向键、Enter、Escape 和鼠标选择。
- 输入为空时展示当前订单、最近选择订单和自然排序靠前的订单。
- 选择订单后保持输入框展示“订单号 · 订单名称”，并触发一次批量甘特读取。

订单候选通过服务端搜索接口获取，不一次性向 DOM 注入全部订单选项。已返回查询结果和最近选择结果保存在前端内存缓存中。

### 4.3 方案对比表

- 第一列表头输出空白单元格，不显示“选”。
- 空白表头内保留仅屏幕阅读器可见的“选择方案”文本。
- 复选框行为、勾选上限 4、颜色联动保持不变。

### 4.4 机器分类利用率表

- 机器类型列目标宽度为 120–140px，按容器宽度允许轻微收缩。
- 机器类型单元格只显示类型名称，不再显示类型 ID、机器总数和“关键”文字。
- 方案列等宽分配剩余空间。
- 方案表头允许自然换行，取消 90px 限制、单行省略和截断，确保完整名称可见。
- 所有方案数据格统一只显示百分比。
- 删除“有排产台数/总台数”和利用率色条。
- 每行最佳百分比仍加粗并沿用最佳值颜色。

后端可以继续返回 `used_machines`、`machines_total` 和 `is_critical`，以保持接口兼容；前端不再展示这些字段。

## 5. 后端读取设计

### 5.1 方案解析顺序

`_resolve_export_solution` 按以下顺序解析方案：

1. 基线方案；
2. `export_result.reference_solutions` 中的完整参考方案；
3. `export_result.solutions` 中的完整优化结果方案；
4. 当前任务参考方案中确认未被截断的完整排程；
5. `RULE:*` 对应的 `RuleReferenceCacheStore` 全量缓存。

任务参考方案可能是面向页面预览的 120 条截断视图，不能覆盖 `export_result` 中的完整方案。评审读取接口不得同步执行 `_rule_reference_solution`。如果 `RULE:*` 没有全量缓存，返回 HTTP 409 和明确的“规则方案尚未计算”错误，由现有规则计算流程生成缓存；不能在甘特或利用率读取期间隐式启动仿真。

### 5.2 惰性评审读取索引

为每个 `(instance_version, task_id, solution_id)` 建立进程内读取索引：

```text
ReviewSolutionIndex
├── solution_id
├── total_operations
├── order_facets
├── entries_by_order: order_id -> schedule entry references
└── machine_type_utilization
```

设计约束：

- 第一次读取方案时单次遍历完整排程，生成订单 facet、订单分组和利用率。
- `entries_by_order` 保存原排程条目的引用，不复制条目字典。
- 后续切换订单直接读取对应列表，不再全量扫描排程。
- 新实例导入、实例版本变化、重新优化或任务被替换时清除对应索引。
- 服务重启后允许首次请求惰性重建，不要求新增持久化表。
- 缓存设置容量上限，只保留最近使用的任务/方案索引，避免长期运行时无界增长。

### 5.3 批量评审读取接口

新增批量评审数据接口：

```text
GET /api/optimize/hybrid/result/{task_id}/review-data
    ?solution_ids=S-1,S-2
    &order_id=O-001
    &include_utilization=true
```

响应结构：

```json
{
  "task_id": "task-id",
  "order_id": "O-001",
  "solutions": ["S-1", "S-2"],
  "schemes": {
    "S-1": [],
    "S-2": []
  },
  "type_utilization": {
    "types": []
  },
  "failed_solution_ids": []
}
```

行为：

- 初次勾选集合变化时，前端发送一次请求，并设置 `include_utilization=true`，同时取得当前订单甘特和利用率。
- 仅切换订单时发送一次请求，并设置 `include_utilization=false`，复用已缓存的利用率。
- 单个方案不可读取时记录到 `failed_solution_ids`，其余方案仍正常返回。
- 方案 ID 去重，最多处理前端允许的 4 个方案。
- 原有单方案 `/schedule` 和 `/machine-type-utilization` 接口暂时保留，避免影响其他页面。
- 新接口使用同步 `def` 端点交给 FastAPI 线程池执行，避免首次索引构建或大响应序列化阻塞事件循环。

### 5.4 订单搜索接口

新增评审订单搜索接口：

```text
GET /api/optimize/hybrid/result/{task_id}/review-orders
    ?solution_ids=S-1,S-2
    &q=001
    &limit=50
```

接口基于选中方案读取索引中的订单 facet 做匹配，返回方案订单集合的并集。结果包含 `order_id`、`order_name` 和可选的工序计数；前端只展示前两个字段。

该接口同样使用同步 `def` 端点；搜索只访问已建索引，首次访问需要建索引时也不会占用事件循环。

## 6. 前端状态与渲染设计

### 6.1 请求状态

评审读取状态拆分为：

```text
reviewSelectionState
reviewOrderSearchState
reviewUtilizationState
reviewScheduleState
```

每类状态使用稳定 key：

- 利用率：`task_id + sorted(solution_ids)`
- 甘特：`task_id + sorted(solution_ids) + order_id`
- 订单搜索：`task_id + sorted(solution_ids) + normalized_query`

### 6.2 取消过期请求

- 每类请求持有独立 `AbortController`。
- 勾选集合、订单或搜索词变化时，立即取消同类旧请求。
- 保留响应 key 校验作为第二层竞态保护。
- 被取消的请求不展示错误提示。

### 6.3 局部刷新

勾选方案时：

1. 立即更新当前行复选框、颜色和已选数量，目标小于 100ms。
2. 仅把利用率区域和甘特区域切换为加载状态。
3. 请求完成后只更新这两个区域。

不再从 `loadReviewGantt`、`loadTypeUtilization` 中调用整页 `renderCurrentPage()`。方案对比表、方案池工具和其他评审内容不因数据加载而重建。

### 6.4 甘特实例复用

评审联动甘特保持稳定 DOM 容器和 Timeline 实例：

- 首次进入时创建 Timeline。
- 方案或订单切换时更新 groups/items DataSet。
- 使用 `setWindow(initialWindow.start, initialWindow.end)` 应用新的 96 小时窗口。
- 离开页面或实例失效时才销毁 Timeline。

空数据、错误状态和部分方案失败时可以暂时隐藏画布，但不因普通加载过程重复创建实例。

### 6.5 前端结果缓存

- 利用率缓存按勾选方案集合保存。
- 甘特缓存按勾选方案集合和订单保存。
- 订单搜索缓存按查询词保存，并保留最近选择订单。
- 切回已经查看的订单时直接渲染缓存数据。
- 新实例、重新优化或任务 ID 变化时清空全部评审缓存。

## 7. 错误处理

- 规则方案未计算：显示“该规则方案尚未生成完整排程，请先计算规则参照”，不自动仿真。
- 部分方案失败：正常展示成功方案，并在甘特上方列出未加载方案。
- 订单无排程：组合框保留订单，甘特区域显示当前勾选方案在该订单下无排产。
- 订单搜索失败：保留当前订单和已加载甘特，不清空页面。
- 请求被新操作取消：静默结束，不覆盖较新的状态。
- 读取索引失效：按实例版本清除并重建一次；重建仍失败则返回明确错误。

## 8. 性能监测与验收

服务端在评审接口记录以下阶段耗时：

- `resolve_solution_ms`
- `build_index_ms`
- `lookup_order_ms`
- `utilization_ms`
- `serialize_ms`
- `total_ms`

前端记录：

- 勾选到视觉反馈时间；
- 请求耗时；
- 数据转换耗时；
- Timeline 更新耗时。

验收标准：

1. 甘特初始可视跨度为 96 小时；完整排程不足 96 小时时显示完整范围。
2. 5000 个以上订单仍能按订单号模糊搜索，候选列表最多渲染 50 条。
3. 方案对比第一列表头无“选”字。
4. 利用率表机器类型列明显收窄，方案名称完整显示，方案数据格只有百分比。
5. 勾选复选框的本地视觉反馈小于 100ms。
6. 4 个方案联动只发送一次批量评审请求。
7. 缓存命中后的订单切换目标小于 300ms。
8. 大实例首次建立单方案读取索引目标控制在 1 秒内，后续复用。
9. 甘特和利用率读取期间 `_rule_reference_solution` 调用次数为 0。
10. 快速连续切换方案或订单时，最终页面只展示最后一次选择的数据。

## 9. 测试设计

### 9.1 后端单元测试

- `RULE:*` 命中规则缓存时返回完整缓存方案。
- `RULE:*` 未命中缓存时返回明确错误，断言没有调用 `_rule_reference_solution`。
- 读取索引只在首次访问时扫描完整排程。
- 同一方案切换多个订单时复用索引。
- 实例版本或任务变化后索引正确失效。
- 机器类型利用率与改造前口径完全一致。

### 9.2 API 测试

- 批量接口一次返回 1–4 个方案的目标订单排程。
- `include_utilization=false` 时不重复计算或返回利用率。
- 单个方案失败时其他方案仍正常返回。
- 订单搜索覆盖精确、前缀、包含、名称匹配、自然排序和 50 条上限。
- 未知订单、未知方案和未计算规则返回可读错误。

### 9.3 前端测试

- 默认窗口为 96 小时并正确处理排程起止边界。
- 订单组合框支持鼠标和键盘操作。
- 快速输入只采用最后一次搜索响应。
- 快速切换订单会取消旧甘特请求。
- 勾选 4 个方案后表头完整、利用率格仅显示百分比。
- 甘特切换使用同一 Timeline 实例，不重复挂载 tooltip 或事件。

### 9.4 性能基准

构造至少包含 5000 个订单、100000 条排程、4 个方案的读取基准，分别测量：

- 首次索引建立；
- 缓存命中读取；
- 订单搜索；
- 订单切换；
- 前端数据转换和 Timeline 更新。

性能基准只验证读取和渲染，不重新执行排产优化。

## 10. 实施边界

本设计不包含：

- 排产算法或仿真内核优化；
- 利用率业务公式调整；
- 方案选择数量从 4 扩大；
- 评审数据持久化表；
- 甘特图编辑、拖拽改排；
- 其他页面的整体框架重构。

实施时应按“规则读取缓存修复 → 后端索引与批量接口 → 前端局部状态与请求取消 → 甘特窗口和搜索组件 → 表格样式 → 性能验证”的顺序推进，每个阶段保持现有接口测试通过。

## 11. Implementation verification（2026-07-19）

验证环境：Apple M4（arm64）、macOS 26.3。墙钟耗时仅用于记录本机结果，性能阈值在基准中保持信息性，不因硬件波动导致失败。

- 完整自动化测试：`261 passed, 35 subtests passed`，2 条既有 FastAPI `on_event` 弃用警告，总耗时 4.04 秒。
- 确定性读取基准：5000 个订单、每单 20 条、每方案 100000 条、40 台机器、4 个方案，共 400000 条排程。
  - 四次冷索引：44.398 / 43.889 / 44.482 / 55.772 ms，中位数 44.440 ms，总计 188.540 ms；低于 1000 ms 信息性目标。
  - 100 次暖订单读取：中位数 0.000083 ms，总计 0.012 ms。
  - 100 次模糊搜索：中位数 2.043 ms，总计 241.275 ms。
  - 四方案目标订单提取：80 条，总计 0.004542 ms。
  - 暖交互组合总计 241.292 ms；低于 300 ms 信息性目标。
- 真实 Chromium 浏览器验收：6/6 检查通过。
  - 方案对比第一列表头可见文本为空，同时保留无障碍名称“选择方案”。
