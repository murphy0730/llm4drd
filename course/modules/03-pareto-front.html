Pareto 前沿：没有唯一最优，只有权衡边界

当"快"和"准时"互相打架时，"最优解"这个词本身需要重新定义。

两个目标，一场拉锯

你想要 Makespan 最小（尽快清完），又想要总延迟最小（别让谁超期）。问题是：为了压低总延迟，你会插队赶那些快到期的活，结果机器切换频繁、Makespan 反而变长。它们常常此消彼长。

这时"哪个方案最好"没有唯一答案。数学上用一个精确概念来判断优劣：支配。

✅
A 支配 B

A 每个目标都 ≥ B，且至少一个严格更好 → B 是纯粹的输，淘汰。

⚖️
A、B 互不支配

A 完工更快但延期更多，B 反过来 → 各有千秋，都得留着。

支配，写成代码就六行

CODE · optimization/solution_model.py 
def dominates(self, other, specs):
    any_better = False
    for spec in specs:
        left  = self.objectives[spec.key]
        right = other.objectives[spec.key]
        if spec.direction == "min":
            if left > right: return False
            if left < right: any_better = True
    return any_better
大白话

逐个目标比较 self（我）和 other（对方）。

读出双方在这个目标上的数值。

对"越小越好"的目标：只要我在任何一项上更差（更大），立刻判负——我没资格支配它。

只要我在某一项上严格更好（更小），就记下"至少赢了一项"。

全程没输、且至少赢一项 → 我支配它。这就是支配的全部定义。

注意 spec.direction：利用率这类目标是"越大越好"，方向相反。

前沿：山脊线，不是山顶

把所有"没被任何方案支配"的解收集起来，就得到 Pareto 前沿。它不是一个点，而是一条边界。

🏔️
登山者的山脊线

前沿像一条山脊线：线上每一点，往任何方向再走都得下坡（牺牲某个目标）。优化的目标不是"登顶单一最高峰"，而是把这条脊线找全、找准——让决策者能在"快"和"准时"之间自由挑一个平衡点。

系统用一个 ParetoArchive 实时维护这条脊线。每来一个新解，就做一次"擂台裁决"：

CODE · optimization/archive.py 
def consider(self, solution):
    if any(e.dominates(solution, self.specs) for e in self._solutions):
        return False
    survivors = [e for e in self._solutions
                 if not solution.dominates(e, self.specs)]
    survivors.append(solution)
    self._solutions = survivors
    return True
大白话 · 擂台裁决

新解上擂台。如果存档里已有任何一个解支配它——它是纯输，直接拒收。

否则，反过来把存档里被这个新解支配的旧解全部踢下擂台。

把幸存的旧解 + 这个新解，组成新的前沿。

返回 True 表示"入选前沿"。（真实代码还会先按排程/目标去重。）

这样存档永远只装着互不支配的解——一条随搜索不断被刷新的脊线。

前沿好不好，怎么量？超体积

两条前沿摆在一起，哪条更优？看谁"罩住"的目标空间更大。这个指标叫 超体积。系统用蒙特卡洛采样来近似它：

1
归一化

把所有目标缩放到 [0,1]，消除量纲差异。

2
随机撒点

在参考立方体里撒 1500 个随机探针。

3
数比例

统计有多少探针被前沿"罩住"，比例 × 立方体体积 ≈ 超体积。

🎯
为什么用采样近似

精确算超体积在高维目标下代价指数级。撒点估比例——用可控的随机误差换来近乎线性的速度，是工程里典型的"够用就好"取舍。

检验一下理解

两目标都"越小越好"。方案 A =(Makespan 100, 总延迟 50)，方案 B =(Makespan 120, 总延迟 40)。谁支配谁？

既然前沿上的解互不支配，系统最终交给排产决策者的应该是什么？
