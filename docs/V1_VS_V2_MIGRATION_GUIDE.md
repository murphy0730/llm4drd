# V1 vs V2 前台对比与迁移指南

## 概览

| 维度 | V1（旧版本） | V2（新版本） | 变化 |
|------|-----------|-----------|------|
| **导航方式** | 5 个水平标签页 | 左侧导航 + 分层菜单 | 从线性流 → 树形结构 |
| **目标用户** | 全部（无差异） | 按角色区分展示 | 领导/计划员/工程师各得其所 |
| **首屏体验** | 工程工作台 | 智能仪表板 | 从配置 → 决策 |
| **页面组织** | 技术模块驱动 | 业务流程驱动 | 问题→洞察→方案→决策 |
| **信息密度** | 单一视角、均匀分布 | 多维视角、层级清晰 | 高价值信息优先展示 |
| **交互模式** | 配置→执行→查看结果 | 引导式逐步完成 | 降低用户认知负荷 |
| **展示与操作** | 在同一页面混杂 | 仪表板（展示） + 工作台（操作）分离 | 各司其职，互不干扰 |
| **右侧栏** | 无 | 场景信息 + 快速帮助 | 增加上下文感知 |
| **后端兼容性** | — | ✓ 100% 兼容 | 零破坏升级 |

## 第一部分：页面对照表

### V1 五个标签页

```
顶部品牌栏 + 英雄区
↓
左侧标签导航（竖直）           右侧内容区
├─ ⭐ 01 问题配置                  [当前选中的页面内容]
├─ 02 异构图
├─ 03 仿真排产
├─ 04 混合优化
└─ 05 大模型配置

右下角：进度标签 (Progress Rail)
```

**特点：**
- 5 个标签页线性排列
- 每个标签对应完整功能模块
- 从上到下依次解锁（前置依赖关系）
- 右侧是单一的内容展示区

│  ├─ 场景库                                            ├─ 优化进度
│  └─ 新建场景                                          └─ 快速帮助

│  ├─ 洞察中心

│  │  ├─ 资源分析
│  │  └─ 瓶颈分析
│  └─ 方案评审
├─ 操作工作台
│  ├─ 工作台(逐步引导)
│  ├─ 问题配置
│  │  ├─ 实例生成与导入
│  │  ├─ 订单维护
│  │  ├─ 资源维护
│  │  └─ 停机与初始状态

- 菜单根据"是否已加载场景"动态显示
- 右侧常驻信息栏增加上下文感知

## 第二部分：核心功能迁移对照

### 问题配置与数据维护

#### V1 路径
```
问题配置页面
├─ 左上：形成生成参数 (12 个字段的网格)
├─ 右上：实例总览 (KPI 卡片)
├─ 中间：机器/工装/人员概览 (3 列卡片)
├─ 中下：停机维护 (表单 + 表格)
├─ 下部：订单维护表
├─ 下部：任务维护表
├─ 下部：工序维护表
├─ 下部：设备班次维护表
└─ 下部：初始在制状态表
```

**问题：**
- 所有内容在一页，用户需要高度滚动
- 配置参数与结果混在一起
- 表格众多，视觉压力大
- 难以追踪当前在做什么

#### V2 路径

**方案 A：场景库 + 新建实例**
```
场景库 (管理多个项目)
  └─ [选择场景] → 自动进入仪表板
  
新建场景 (选择方式)
  ├─ [快速生成] → 参数表单 (简化为 6 个关键字段)
  └─ [导入 Excel] → 文件上传

工作台 Step 1: 问题设计
  └─ 编辑器 (参数修改、模板应用)

工作台 Step 2: 约束校准
  ├─ 订单维护 → 弹窗表格编辑器
  ├─ 工序维护 → 弹窗表格编辑器
  ├─ 资源维护 → 独立模块
  ├─ 班次管理 → 独立模块
  └─ 停机维护 → 独立模块
```

**优势：**
- 配置流程清晰（新建 → 设计 → 校准）
- 表格编辑器支持"内联编辑"与"弹窗模态"两种模式
- 每个维护功能独立，避免信息过载
- 支持多个场景同时存在，快速对比

### 异构图分析

#### V1 路径
```
异构图页面
├─ 左侧：构建图 (按钮)
├─ 左侧：节点类型筛选 (Checkboxes)
├─ 左侧：边类型筛选 (Checkboxes)
├─ 左侧：关键字搜索 (Input)
├─ 中间：图形化展示 (Cytoscape/Sigma)
├─ 下方：节点详情面板
└─ 下方：边类型统计
```

**问题：**
- 控制项分散在左右两侧
- 图形展示与表格信息重叠
- 难以全屏查看大规模图

#### V2 路径

**方案 A：工作台 Step 3 内联**
```
工作台第 3 步 (结构与仿真)
├─ 左侧工作区：构建异构图
│  └─ 参数配置 + 运行按钮 + 即时结果
└─ 右侧工作区：规则仿真  
   └─ 规则选择 + 运行 + 结果展示
```

**方案 B：洞察中心深度分析**
```
决策 → 洞察中心 → 结构分析
├─ Tab 1: 拓扑关系 (订单→任务→工序)
├─ Tab 2: 关键路径分析
├─ Tab 3: 瓶颈识别
└─ Tab 4: 资源冲突图
```

**优势：**
- 图分析与仿真结合，相互印证
- 逐层递进深度分析，避免一次性信息过载
- 支持导出分析报告

### 仿真排产

#### V1 路径
```
仿真排产页面
├─ 左侧：规则选择 (Select Box)
├─ 中间：KPI 卡片 (3-5 个关键指标)
├─ 中间：甘特图 (日期时间坐标)
└─ 下方：排程明细表
```

**问题：**
- 参数选择简陋
- 多个规则需要多次切换标签
- 甘特图与表格不同步

#### V2 路径

**工作台 Step 3 (结构与仿真)**
```
左侧卡片：参数配置
├─ 规则单选 (ATC/EDD/SPT/LPT/CR/FIFO/MST/...)
├─ 并行对比选项 (Multi-Select, 同时运行多个规则)
└─ [运行] 按钮

中间区：即时结果面板
├─ KPI 卡片 (完工时间、延期、利用率)
├─ 规则对比柱状图 (A/B/C 规则的目标函数值)
└─ 甘特图预览 (可展开查看详情)

右侧卡片：详情钻取
├─ 【点击甘特中的任务】显示
  - 任务详情（订单、优先级、交期）
  - 工序序列
  - 机器选项
  - 工装要求
  - 人员技能
```

**优势：**
- 支持批量规则对比，一次运行获得多个结果
- 参数、结果、详情在同一工作区
- 可快速评估多个规则的相对好坏

### 混合优化与方案评审

#### V1 路径
```
混合优化页面 (融合 4 个子功能)
├─ 上：目标选择 (Checkboxes × 多个目标)
├─ 上：参数详细配置 (~15 个参数)
├─ 中：基线规则选择与运行
├─ 中：启动优化按钮 + 进度条
├─ 中：Pareto 方案库表格 (所有候选)
├─ 下：启发式参考规则运行结果
├─ 下：精确参考解 (OR-Tools)
├─ 下：AI 评审对话框
└─ 下：方案导出
```

**问题：**
- 内容过多，逻辑混杂
- 优化配置、执行、结果、评审混在一页
- 初用者不知道先做什么
- 方案对比没有重点

#### V2 路径

**方案 A：工作台 Step 4 (多目标优化)**
```
左侧卡片：参数配置
├─ 目标选择 (1-5 个)
├─ 优化策略 (快速/均衡/深度 预设)
├─ 自定义参数 (可选，高级用户用)
└─ [启动优化] 按钮

中间区：进度与结果
├─ 实时进度条 (已测试 X 个候选，当前最优 Y)
├─ Pareto 前沿曲线 (动态更新)
└─ 候选方案列表 (前 10 个)
```

**方案 B：工作台 Step 5 (方案评审)**
```
方案库表格 (所有 Pareto 候选)
├─ 列：方案ID、关键指标、推荐度、操作

对比工具
├─ [多选] 方案 A、B、C
├─ [对比] 按钮
└─ 详细对比面板（并排展示）
  - 甘特图对比
  - KPI 对比柱状图
  - AI 评价

方案详情面板
├─ 【点击行】显示
  - 机器占用情况
  - 人员分配详情
  - 工装复用频次
  - 关键路径
  - AI 推荐理由
```

**方案 C：决策 → 智能仪表板**
```
候选方案对比表 (Top 5)
├─ 推荐排序
├─ 关键指标
├─ 推荐度星级
└─ [详细评审] 进入 Step 5 深度分析
```

**优势：**
- 优化参数配置 → 执行进度 → 方案评审 分离明确
- 方案对比有专门页面，避免混杂
- 多维度评价（指标 + AI 推荐 + 可视化对比）

## 第三部分：重要变化点

### 变化 #1：导航方式激变

**V1：** 5 个标签页，线性排列，用户从上往下逐个探索

```
[01] [02] [03] [04] [05]
 ↓
点击第 n 页 → 加载第 n 页内容
```

**V2：** 树形菜单，并且会根据"是否已加载场景"动态显示

```
场景
├─ 场景库 ← 应用启动时首先看到
└─ 新建场景

决策路径 ← 【只有在加载场景后才显示】
├─ 智能仪表板
├─ 洞察中心
└─ 方案评审

工作台 ← 【只有在加载场景后才显示】
├─ 工作台(逐步引导)
├─ ...
```

**用户心智变化：**
- V1：用户需要理解"这 5 个页面的依赖关系"
- V2：用户只需"选择场景 → 看仪表板 → 进工作台"

---

### 变化 #2：信息架构重组

**V1：** 按技术模块组织

```
问题配置 ← 所有的"配置"工作
异构图   ← 所有的"分析"工作
仿真     ← 所有的"仿真"工作
优化     ← 所有的"优化"工作
LLM      ← 所有的"配置"工作（第二次）
```

**问题：** 同样的配置工作（问题配置 + LLM 配置）散落在两个标签页

**V2：** 按业务流程 + 用户角色组织

```
场景
  └─ 管理多个实例

决策路径
  ├─ 给领导看（一屏决策）
  │  └─ 智能仪表板
  ├─ 给工程师看（深度分析）
  │  └─ 洞察中心（结构 + 资源 + 瓶颈）
  └─ 给团队看（方案对比）
     └─ 方案评审

工作台
  ├─ 给计划员（问题配置）
  │  └─ Step 1-2
  ├─ 给工程师（仿真与分析）
  │  └─ Step 3
  └─ 给优化工程师（参数调优）
     └─ Step 4-5

系统
  └─ 给管理员（配置管理）
```

### 变化 #3：首屏体验转变

**V1：** 打开应用 → 看到"问题配置"的表单和表格

```
用户反应：
"哦，这是一个配置工具，我需要先填表..."
```

**V2：** 打开应用 → 看到"场景库"（如无场景）或"智能仪表板"（如有场景）

```
有场景时：
  用户反应：
  "哦，这是一个决策支持系统，我看一下推荐方案..."
  
无场景时：
  用户反应：
  "我需要先创建或导入一个场景"
  （明确的下一步）
```

**心智优化：** 从"工具"思维 → "决策"思维

---

### 变化 #4：分离"展示"与"操作"

**V1：** 一切都混在一起

```
问题配置页面
├─ 第一屏：配置参数 (表单)
├─ 第二屏：实例概览 (KPI)
├─ 第三屏：表格编辑
├─ 高度滚动...
```

**问题：**
- 领导看不到"一屏的决策支持"
- 计划员配置时被冗余信息干扰
- 无法快速"展示方案"给决策者

**V2：** 明确分离

```
展示路径（领导看）
  └─ 仪表板：问题 + 洞察 + 方案 + 建议
     （完全不涉及参数配置）

操作路径（计划员与工程师看）
  ├─ 工作台：逐步完成任务
  └─ 可自由切换回仪表板快速查看

配置路径（少数管理员用）
  └─ 系统 → 配置管理
```

## 第四部分：用户迁移指南

### 4.1 对于现有 V1 用户

**最小学习成本方式：**

1. **首次打开 V2**
   - 点 [场景库]
   - 点 [新建场景]
   - 选择 [快速生成] 或 [导入 Excel]
   - **这和 V1 的"问题配置"完全一样**

2. **加载场景后**
   - 自动进入仪表板
   - **这是新功能**，快速看一下（推荐给谁看什么）

3. **需要编辑约束时**
   - 点左侧 [工作台]
   - 点第 2 步 [约束校准]
   - **这和 V1 的几个"编辑表格"完全一样**

4. **运行仿真**
   - 工作台 Step 3
   - **和 V1 完全一样，但现在支持批量运行多个规则**

5. **启动优化**
   - 工作台 Step 4
   - **参数配置简化了，但支持高级自定义**

6. **查看方案**
   - 工作台 Step 5，或
   - 决策 → 方案评审
   - **两条路都行，后者更专业**

**迁移难度：** ⭐ 低（一周内熟悉）

### 4.2 对于技术架构师

**后端 API 是否需要改？**

✓ **不需要改。** V2 前端 100% 兼容现有 API。

**是否需要添加新的后端功能？**

**可选：**
- 场景管理接口（保存、加载、删除、共享）
  - 不添加也行，现在是本地管理 + 记忆化
- 场景导出为 Excel 模板
  - 现在只支持"导入"，不支持"导出当前场景为模板"
  - 可选完善

**不需要改的：**
- `/api/instance/*`
- `/api/simulate`
- `/api/optimize/*`
- `/api/graph/*`
- `/api/llm-config`

### 4.3 对于产品与设计

**用户反馈关键指标：**

- **首屏加载时间** 目标 <2s
- **点击执行任务的时间** 目标 <500ms
- **用户首次使用" Learning Curve"** 目标 <10 分钟
- **领导汇报时长** 目标 5-10 分钟（VS 30 分钟的 V1）

**A/B 测试建议：**

1. **灰度发布** 10% 用户尝试 V2，收集反馈
2. **关注以下指标：**
   - 用户停留时长（仪表板吸引度）
   - 工作台完成度（流程合理性）
   - 错误率（易用性）
3. **收集定性反馈：**
   - "相比 V1，最喜欢的功能是？"
   - "最不喜欢的功能是？"
   - "缺少什么功能？"

## 第五部分：常见问题 FAQ

### Q1: V2 会不会太重了？加载时间会不会很长？

**A:** 不会。

- index_v2.html: 15 KB（压缩后 4 KB）
- app_v2.css: 25 KB（压缩后 8 KB）
- app_v2.js: 35 KB（压缩后 12 KB）

**总计：** 75 KB（压缩后 24 KB），和 V1 规模接近。

**加载优化：**
- 页面按需加载（非活跃页面的 DOM 不初始化）
- 图表库（ECharts）按需动态加载
- 场景数据走本地缓存（IndexedDB）

---

### Q2: V2 能不能和 V1 同时存在？

**A:** 可以。

```
http://localhost:8000       → V1 (index.html)
http://localhost:8000/v2    → V2 (index_v2.html)
```

用户可以自由选择使用哪个版本。推荐灰度发布时采用这种策略。

---

### Q3: V2 对浏览器版本有要求吗？

**A:** 支持所有现代浏览器。

- Chrome 90+
- Firefox 88+
- Safari 14+
- Edge 90+
- IE 11：**不支持** （使用了 ES6 语法）

---

### Q4: 如何从 V1 迁移已有的"收藏夹场景"或配置？

**A:** 两种方式：

**方式 1：导出 Excel**
- V1 中点 [导出 CSV] 或 [模板导出]
- V2 中点 [导入 Excel]
- ✓ 完整迁移

**方式 2：本地保存**
- V1 → 浏览器开发者工具 → Application → LocalStorage
- 复制 `llm4drd_*` 的 key-value
